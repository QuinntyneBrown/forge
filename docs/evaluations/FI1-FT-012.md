### Mid-implementation finding (resolved before slice closure)

**FT-004 `FieldComponent` does not work with `<input matInput>` content projection.** The original FT-004 slice description planned to wrap `<mat-form-field>` and project `<input matInput>` via `<ng-content>`. At runtime this throws `Error: mat-form-field must contain a MatFormFieldControl.` because `MatFormField`'s `@ContentChild(MatFormFieldControl)` does not resolve the projected `MatInput` instance through the wrapper component's view. Importing `MatInputModule` in `FieldComponent` does not help — the issue is the projection-based discovery, not directive scoping.

**Resolution.** Removed the `FieldComponent` files and drop the `'./lib/field/field.component'` re-export from `components/public-api.ts`. `SignUpFormComponent` uses `<mat-form-field appearance="outline">` + `<input matInput>` inline. FT-004's task entry in `docs/plans/frontend-tasks.md` is amended to "⚠ design pending" with two viable revised designs (input-as-`@Input()` or directive-based) — the next form-bearing slice (likely `ProfileForm` in FT-031 or `WorkoutDetailForm` in FT-028) will revisit and ship the working pattern.

This deferral does not block any L2 — `MatFormField` is the underlying primitive Implementation Guidance asks for; the wrapper is a project-internal abstraction whose timing is flexible.
