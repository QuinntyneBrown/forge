### Structural checks (criteria 1, 6 — Guidance adherence + SOLID/CQS shape)

- **`ConfirmPasswordResetCommand : IRequest`** (void result; mutation). Colocated `ConfirmPasswordResetCommandValidator` enforces:
  - `Token` non-empty.
  - `NewPassword` non-empty + `MinLength(12)` + complexity regex matching the backend's existing `RegisterCommandValidator` policy (uppercase, lowercase, digit, symbol). The two policies stay aligned by virtue of being copied; if a future slice introduces a shared `PasswordPolicyValidator`, both call sites can adopt it without changing observable behavior.
- **`ConfirmPasswordResetCommandHandler`** depends on `IAppDbContext`, `IPasswordHasher`, `IRefreshTokenStore`, `IClock`, `IAuditLogger`. Flow:
  1. Hash the presented token with SHA-256 (same scheme as the request leg from BT-004).
  2. Look up `PasswordResetTokens` by hash.
  3. If not found, already consumed, or expired → audit `password-reset.failure` and throw `InvalidPasswordResetTokenException` (mapped to 400 by middleware).
  4. Look up the `User`; if missing → same failure handling.
  5. `user.PasswordHash = _hasher.Hash(request.NewPassword)`; `token.ConsumedAt = _clock.UtcNow`. Single `SaveChangesAsync` persists both — atomic from EF's perspective.
  6. Call `IRefreshTokenStore.RevokeAllForUserAsync(user.Id, ct)` so every active refresh-token family for the user is invalidated (L2-033 + L2-004 ac 3).
  7. Audit `password-reset.success`.
- **`IRefreshTokenStore.RevokeAllForUserAsync(userId, ct)`** — new abstraction method, implemented in `RefreshTokenStore` by stamping `RevokedAt = _clock.UtcNow` on every active row for the user. Uses `IClock` (BT-006a seam).
- **`InvalidPasswordResetTokenException`** — new exception type in `Forge.Application.Auth/`. `ExceptionHandlingMiddleware` maps it to `400 Bad Request` with `application/problem+json` body and a generic title (no detail about *why* the token was invalid — keeps the response shape identical for unknown / consumed / expired cases, consistent with the request leg's no-enumeration posture).
- **`AuthController`** adds `[HttpPost("password-reset/confirm")]` (anonymous) — accepts `ConfirmPasswordResetRequest` and returns `204 No Content` on success.

### Acceptance test (criterion 8 — ATDD evidence)

`tests/Forge.Acceptance/Auth/PasswordResetConfirmAcceptanceTest.cs` was authored before the implementation. Header: `// Traces to: BT-005, L2-004, L2-031, L2-033`. Three scenarios:

1. **Happy path.** Register → request reset → capture the raw token via `CapturingPasswordResetEmailSender` (a test seam that swaps the production `LoggingPasswordResetEmailSender` via `ConfigureTestServices`) → confirm with a new password → 204. Asserts old password no longer signs in (401), new password signs in (200), and the pre-confirm refresh token is revoked (`POST /api/auth/refresh` → 401).
2. **Reuse.** Register → request → confirm with a new password → 204 → confirm again with the *same* token and a different new password → 400. Asserts the password didn't silently change to the third value: signing in with that third value returns 401 while the second value still works.
3. **Expiration.** Register → request → advance the `FakeClock` past the 30-minute TTL → confirm → 400. Asserts the original password still works (200).

The `CapturingPasswordResetEmailSender` is a test-only `IPasswordResetEmailSender` registered via `ConfigureTestServices`. It overrides the production `LoggingPasswordResetEmailSender` in the test host's DI graph — production binaries don't carry the capturing sender, so this stays purely a test seam.

### Non-blocking observations

- Both `password-reset.success` and `password-reset.failure` audit rows are written per BP1 §6 / §10. The failure row carries the `userId` of the looked-up token (when the token resolved at all) so a security-team review can correlate per-user attempt rates without reading the raw token. The audit logger's redaction (BT-008) ensures no token / hash / password value reaches the row.
- `InvalidPasswordResetTokenException` returns the same response shape regardless of why the token was invalid (unknown / consumed / expired). That's intentional — leaking *why* would be an enumeration channel.
- `RevokeAllForUserAsync` is also useful for BT-006 (account deletion) — it can call the same method to revoke tokens before anonymizing the user. Calling out so the BT-006 implementer reuses the abstraction rather than re-inventing it.
- The `password-reset.failure` audit on the unknown-token path uses `userId: null` (because `token` is null when the hash doesn't match anything in the table). The path matters for L2-035: even random/guessed token attempts get recorded for security review; we don't drop them on the floor.
- `dotnet test` total time crept from ~17s (BT-007 era) to ~26s with this slice — the three new scenarios each spin up a fresh LocalDB. Acceptable; if it grows past ~60s the BT2 non-blocking note about a shared base factory class becomes worth acting on.
