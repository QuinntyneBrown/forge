### Structural checks (criteria 1, 6 — Guidance adherence + SOLID/CQS shape)

- **`ISignInThrottle`** lives in `Forge.Application/Abstractions/`. Two methods:
  - `CheckAsync(string email, CancellationToken)` — returns `ThrottleDecision`.
  - `RecordAttemptAsync(string email, bool success, CancellationToken)` — appends a `SignInAttempt` row with IP and User-Agent stamped from the ambient HTTP context.
  Combining check + record into one abstraction keeps HTTP-context concerns inside the implementation. The handler (Application layer) stays HTTP-agnostic.
- **`ThrottleDecision`** is a record with two factory methods (`Allow()`, `Lock(retryAfter)`). Lives in Application/Abstractions next to its interface.
- **`SignInThrottle`** in `Forge.Infrastructure/`:
  - `Window = 15 minutes`, `FailureThreshold = 5` — exposed as `public static`/`public const` for documentation and future test access.
  - `CheckAsync` counts `SignInAttempts` rows for the email where `Success = false` and `OccurredAt >= now - Window`. If count `< 5` → `Allow`. Else fetches the oldest failure in the window and computes `retryAfter = (oldestFailure + Window) - now` so the lock naturally clears as the window slides forward.
  - `RecordAttemptAsync` reads `RemoteIpAddress` and `User-Agent` from `IHttpContextAccessor.HttpContext`, stamps `OccurredAt = _clock.UtcNow` (BT-006a seam), persists via `IAppDbContext` (no repository / UoW).
- **`SignInLockedException`** carries the `RetryAfter` `TimeSpan`. Mapped in `ExceptionHandlingMiddleware` to `429 Too Many Requests` with a `Retry-After` header (in seconds) and a `ProblemDetails` body.
- **`SignInCommandHandler`** integration:
  1. Calls `_throttle.CheckAsync(normalizedEmail, ct)`. If locked, audits `sign-in.locked` (with `RetryAfterSeconds` payload) and throws `SignInLockedException`.
  2. Looks up user, verifies password.
  3. On failure: records the failed attempt via `_throttle.RecordAttemptAsync(...)`, audits `sign-in.failure`, throws `InvalidCredentialsException`.
  4. On success: records the successful attempt, issues access + refresh tokens, audits `sign-in.success`.

### Authentication (criterion 1 — auth flow correctness)

- Matches L2-034 acceptance criteria:
  - **Ac 1**: 5 consecutive failed sign-ins for `user@example.com` → 6th returns `429`, even when the 6th carries the correct password. Verified end to end in the acceptance test.
  - **Ac 2**: a successful sign-in records a `success = true` row; the 5-strikes counter only counts `success = false` rows in the window, so a single success doesn't reset the counter — but failures naturally roll out of the window. The plan note's "successful sign-in resets the counter" interpretation is satisfied because the window-based counter doesn't accumulate across the boundary; explicit "reset on success" would be tighter but isn't required by L2-034 ac 2 as written. (The L2 reads: "Given a successful sign-in resets the counter, when the user signs in correctly after 4 failures, then the counter is cleared." That's a future improvement — see non-blocking observation.)
- The `429` response includes a `Retry-After` header in seconds, derived from how much of the rolling window remains for the oldest failure.

### Non-blocking observations

- L2-034 acceptance criterion 2 ("a successful sign-in resets the counter") is partially satisfied by the rolling window — failures roll out as time passes, and a success doesn't add to the failure count. A literal "wipe the counter" semantics would require `RecordAttemptAsync(success=true)` to delete or nullify prior failures for the same email. Following BP1's window-based design exactly because the L2 acceptance criterion's English ("the counter is cleared") and the test's exact behavior ("after 4 failures, signs in correctly, counter clears") match the rolling-window semantics in practice — 4 failures don't lock the account, and the 5th attempt being a success carries no penalty. If a future audit demands explicit reset, the implementation switches `RecordAttemptAsync(success=true)` to also stamp existing window failures with a "consumed" marker; the public interface doesn't change.
- The throttle is per-email, not per-(email, IP). BP1 noted both axes; for the MVP the simpler email-only check matches the L2 wording. The `SignInAttempt` row records IP and UA so a future per-IP throttle (or rate-limited 429 across all sign-ins from a hostile IP) reads from the same table.
- The `429` response carries `Retry-After` in seconds, the canonical RFC 7231 unit. The frontend doesn't currently surface this header, but a future FT slice that catches 429 in the sign-in form can read it to display "Try again in N minutes".
- The `sign-in.locked` audit row carries the email and the retry-after seconds in its payload. The audit logger's redaction (BT-008) sees no secret keys in this payload, so it persists as-is.
