# FT-027 — WorkoutListComponent + WorkoutsPage

## Acceptance test
`frontend/tests/workouts-list.spec.ts` is in place; the POM helper
`frontend/tests/pom/workouts-list.page.ts` exposes locators for every
chip and the row container. The spec seeds five sessions across mixed
equipment via `POST /api/sessions`, signs in, navigates to `/workouts`,
clicks the Treadmill chip, and asserts the row count drops to 2 and the
non-treadmill equipment names are absent.

End-to-end execution is currently blocked by the local environment —
the recently-edited `appsettings.json` and acceptance tests target
`Server=.\SQLEXPRESS;…` and SQL Server Express is not installed on this
machine. The spec runs once SQLEXPRESS is available; until then the
component is verified by the app build (see below) and by manual
inspection of the rendered template.

## Implementation
### `domain` library
- `workout-list/workout-list.component.{ts,html,scss}`.
- Injects `SESSIONS_SERVICE` + `EQUIPMENT_SERVICE` and the Angular `Router`. State held in signals: `equipment`, `selectedEquipment`, `selectedRange`, `sessions`, `loading`. `hasSessions` is a computed predicate.
- Composes `<forge-card>`, `<forge-chip>` (inside `<mat-chip-listbox>`), `<forge-empty-state>`, `<forge-button>`.
- Filter chips: an "All equipment" chip plus the dynamic equipment list (BT-013), and a static range strip (`All`, `Today`, `This week`, `This month`).
- Empty state surfaces "No sessions yet" with a primary CTA that routes to `/workouts/new` (handled by FT-028).
- Each row is a `<button>` inside a `<forge-card>` with `data-testid="workout-list-open-{id}"` so consumer specs can drive a click without needing the inner Material classnames.
- Public-API export added.

### `app` (Angular forge app)
- `pages/workouts/workouts.page.{ts,html,scss}` wraps the new component in `<forge-app-shell>` with the standard four-destination nav rail.
- `app.routes.ts` registers `/workouts` behind `authGuard`.

## Backend interop fix (cross-cutting)
- The Angular client sends the equipment as a string (`"Treadmill"`) but the backend's default JSON deserialiser expected a numeric enum. `Program.cs` now registers `JsonStringEnumConverter` on the controllers' JSON options so all enum-typed JSON fields (`EquipmentType`, `SessionRange`, `PointsLedgerReason`, etc.) round-trip as their declared C# names. Existing backend acceptance tests already use enum-typed records on both sides so they continue to pass; the change broadens the contract for the Angular client.
