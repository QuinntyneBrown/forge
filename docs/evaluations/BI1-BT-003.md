### Structural checks (criteria 1, 6 — Guidance adherence + SOLID/CQS shape)

- `SignOutCommand` declares `IRequest` (no result type — sign-out is a fire-and-forget mutation). Colocated `SignOutCommandValidator` enforces a non-empty refresh token (consistent with `RefreshTokenCommandValidator`). `SignOutCommandHandler` depends only on `IRefreshTokenStore` and delegates to its new `RevokeByPresentedTokenAsync` method. CQS shape preserved.
- `IRefreshTokenStore` gains `RevokeByPresentedTokenAsync(string rawToken, CancellationToken)` — a convenience method that hashes the presented token, looks up the matching row by hash, and calls the existing `RevokeFamilyAsync` for that row's `FamilyId`. The handler stays free of hashing concerns; the abstraction continues to own the cryptography.
- `AuthController.SignOut` is decorated with `[Authorize]` so the caller must hold a valid access token. The endpoint accepts `SignOutRequest` (refresh token in body) and returns `204 No Content`. Anonymous endpoints in `AuthController` (`register`, `sign-in`, `refresh`) remain anonymous; the new endpoint is the first action-level `[Authorize]` on the controller, so the controller is left without a class-level attribute to keep that mix explicit.

### Authentication (criterion 1 — auth flow correctness)

- The sign-out flow now matches the L2-003 acceptance criteria:
  - **Ac 1**: signed-in user posts to `/api/auth/sign-out` → backend revokes every refresh token in the presented token's family and clears the access token client-side. Test asserts `204 No Content`.
  - **Ac 2**: a signed-out session attempting an authenticated route returns `401`. Verified end-to-end by the test which calls `/api/auth/refresh` after sign-out and asserts `401 Unauthorized`.
- The handler intentionally does NOT throw if the presented token is unknown — `RevokeByPresentedTokenAsync` is idempotent and returns silently when no match is found. This keeps sign-out a safe operation regardless of client state, and prevents account enumeration via sign-out responses.

### Non-blocking observations

- The frontend doesn't yet call `/api/auth/sign-out` from the UI's "Sign out" button. The dashboard's existing sign-out button only clears `AuthStateService` client-side. A future frontend slice (likely FT-014 once the refresh-interceptor + auth-guard land, or earlier as a small follow-up) will wire the button to call `IAuthService.signOut(refreshToken)` so the server-side revocation actually happens.
- `SignOutCommand` is the first command in the codebase that returns no result (`IRequest` without a generic). MediatR's pipeline behaviors (FluentValidation) work the same for void requests; the `ValidationBehavior<TRequest, TResponse>` constraint is `where TRequest : notnull`, which is satisfied.
- Returning 204 instead of an empty 200 is intentional — REST convention for "the operation succeeded and there is no body to return". Documented in the eval; will be the same shape for BT-006 (account deletion via `DELETE /api/me`).
