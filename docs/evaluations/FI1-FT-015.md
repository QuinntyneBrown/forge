### Implementation

`CheckboxComponent` (in `components/`):
- Wraps `<mat-checkbox [checked] [disabled] (change)="onChange($event)">` with `[attr.data-testid]="testid"` on the host (Material's mat-checkbox doesn't expose a clean inner input for testid forwarding; the host carries it and the test uses `.click()` rather than `.check()`).
- Inputs: `[checked]`, `[disabled]`, `[testid]`. Output: `(checkedChange)` — emits the new boolean.
- BEM hooks: `checkbox`, `checkbox__label` (the latter on the projected `<ng-content>` slot).

`AuthStateService` (in `forge/`):
- Now injects `IAuthService` (via `AUTH_SERVICE` token) and exposes `tryHydrate(): Promise<void>`.
- `setSession(result, persist = false)` — when `persist=true`, writes `result.refreshToken` to `localStorage['forge.auth.refreshToken']`. Wrapped in try/catch so private-mode / quota-exceeded scenarios don't break sign-in.
- `clear()` removes the persisted entry alongside the in-memory session — sign-out and refresh-failure paths inherit the cleanup automatically.
- `tryHydrate()` reads the persisted refresh token; if present, calls `IAuthService.refresh(persisted)`. On success: `setSession(rotated, persist=true)` (re-rotates the persisted token). On failure: `clear()` (drops the stale entry and routes through normal unauth flow).
- `provideAppInitializer(() => inject(AuthStateService).tryHydrate())` in `app.config.ts` runs hydration during bootstrap. Angular awaits the returned promise before rendering, so `authGuard` sees the hydrated session on first navigation — no flicker.

`SignInFormComponent` (in `domain/`):
- Adds a private `rememberMe` field, a `<forge-checkbox>` with `(checkedChange)="onRememberMeChange($event)"` and `testid="sign-in-remember-me"`, and emits `SignedInEvent { result, rememberMe }` instead of just `result`.

`SignInPage` (in `forge/`):
- Updated to consume `SignedInEvent` and call `auth.setSession(event.result, event.rememberMe)`.

### Acceptance test (criterion 8 — ATDD evidence)

`tests/sign-in-remember-me.spec.ts` was authored before the implementation. Header: `// Traces to: L2-002 (ac 2 — Remember me survives browser restart), L2-033`. Two scenarios:

1. **Checked → persists.** Register, sign in with `getByTestId('sign-in-remember-me').click()` to enable Remember me, dashboard loads. Capture `context.storageState()` (which includes `localStorage`). Open a fresh context with `storageState`. Navigate to `/dashboard` — `provideAppInitializer` triggers `tryHydrate`, which calls `/api/auth/refresh` with the persisted token, gets a rotated pair, populates `AuthStateService`. The auth guard then permits the route, and the dashboard greeting reads the same email. Asserts `dashboard.greeting` contains the registered email.
2. **Unchecked → does not persist.** Same flow without the click. Fresh context's `localStorage` is empty, `tryHydrate` no-ops, auth guard redirects to `/sign-in`. Asserts the URL becomes `/sign-in`.

Both scenarios pass against the live backend, exercising the BT-002 refresh endpoint end to end (rotation observed: the refresh-token persisted into the second context is replaced by a fresh rotated token after `tryHydrate`).

### Mid-implementation finding (resolved before slice closure)

Initial test used `getByTestId('sign-in-remember-me').check()`. Playwright's `.check()` requires a native `<input type="checkbox">` element; Material's `<mat-checkbox>` host wraps an internal input but the data-testid is on the wrapper. **Resolution:** switched to `.click()`, which simulates a user click on the wrapper and triggers the same internal toggle. Documented in this eval as the pattern `<forge-checkbox testid="…">` consumers should use in tests.

### Non-blocking observations

- The L2-002 acceptance criterion 2 doesn't specify how long the persisted session should live. Rotation happens on every hydration, so the effective lifetime equals the BT-002 refresh-token lifetime (14 days). If a future requirement caps Remember-me persistence at a shorter window, the implementation can read a `forge.auth.persistedAt` timestamp alongside the token and discard it past TTL.
- `tryHydrate` runs on every app-bootstrap. The cost is one synchronous `localStorage.getItem` plus, when a token is present, one `POST /api/auth/refresh` round-trip. On cold load this delays render by the round-trip duration (~50-200ms locally). Acceptable trade-off for L2-002 ac 2; if the cost becomes visible in production, the bootstrap could rely on the stored access-token freshness too (out of scope here).
- The `<forge-checkbox>` host carries `data-testid` rather than the inner Material input. This differs from `<forge-button>` which forwards `[attr.data-testid]` to the inner button. Material's mat-checkbox doesn't expose its internal input's host attributes for clean forwarding; using the wrapper + `.click()` is the simplest reliable test target. If a future slice needs to query the inner input directly (e.g., for `:checked` state assertions via attribute selectors), `<forge-checkbox>` can be expanded then.
- `provideAppInitializer` is the Angular 18+ preferred API replacing the older `APP_INITIALIZER` token. Angular 21 supports it directly.
