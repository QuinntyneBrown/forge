# FT-030 — RewardsCatalogComponent + RewardsPage

## Acceptance test
`frontend/tests/rewards-redeem.spec.ts` seeds a 625-min mid-day NY
session to land exactly +1250 base pts, signs in, opens `/rewards`,
asserts the balance label reads 1250, redeems the Smoothie (200 pts),
asserts the balance label flips to 1050. Then drives the
INSUFFICIENT_POINTS branch by posting against the most expensive
reward via the API (the redeem button is disabled for unaffordable
rewards in the UI, so the negative path is asserted at the API
boundary). Same SQLEXPRESS-pending caveat as the previous workouts
specs.

## Implementation
### `api` library
- `models/reward.model.ts` — `Reward { id, name, description, costPoints, sortOrder }` + `RedemptionResult { redemptionId, remainingBalance }` matching BT-026 / BT-027.
- `IRewardsService` extended with `listRewards(): Observable<Reward[]>` and `redeem(rewardId): Observable<RedemptionResult>`.
- `RewardsService` impl: `GET /api/rewards`, `POST /api/rewards/{id}/redeem`.
- Public-API exports added.

### `domain` library
- `rewards-catalog/rewards-catalog.component.{ts,html,scss}`:
  - Injects `IRewardsService` and `IDashboardService` (for current balance).
  - State held in signals; computed `canAfford(reward)` and `progressToReward(reward)`.
  - Composes `<forge-card>`, `<forge-button>`, `<forge-progress-ring>` (per-row 0..100% fill against the cost).
  - Redeem button disabled when not affordable or another redemption is in flight; loading spinner on the active reward.
  - `INSUFFICIENT_POINTS` from the backend is surfaced as a friendly inline error banner; other errors fall back to a generic message.
- Public-API export added.

### `app`
- `pages/rewards/rewards.page.{ts,html,scss}` composes `<forge-app-shell>`, `<forge-tier-card>`, `<forge-rewards-catalog>`. Route `/rewards` registered behind `authGuard` so the BottomNav `Rewards` destination resolves.

## Library placement
- Service in `api` with contract + token. ✅
- Catalog in `domain`. ✅
- Page in `app`.
