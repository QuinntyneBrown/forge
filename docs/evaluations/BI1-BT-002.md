### Structural checks (criteria 1, 6 — Guidance adherence + SOLID/CQS shape)

- `IRefreshTokenStore` lives in `Forge.Application/Abstractions/`. Concrete `RefreshTokenStore` lives in `Forge.Infrastructure/`. Dependency direction respected (handlers reach for the abstraction; concrete is wired in `DependencyInjection.cs`).
- `RefreshTokenStore` implementation:
  - Generates a 32-byte cryptographically random raw token (`RandomNumberGenerator.GetBytes(32)`), serialized as base64-url (no padding).
  - Persists the SHA-256 hex hash only (`Convert.ToHexString(SHA256.HashData(...))`). The raw token is returned to the caller exactly once.
  - `IssueAsync(userId, familyId, ct)` allocates a new family if `familyId` is null; otherwise carries the existing family forward.
  - `ConsumeAsync(rawToken, ct)` looks up by hash, refuses revoked / expired tokens, **revokes the entire family if the token is already consumed** (replay detection), otherwise marks the row consumed and returns it.
  - `RevokeFamilyAsync(familyId, ct)` stamps `RevokedAt = UtcNow` on every member of the family that's not already revoked.
  - All persistence flows through `IAppDbContext` (no repository / unit-of-work).
- `RefreshTokenCommand` + colocated `RefreshTokenCommandValidator` (FluentValidation: token non-empty) + `RefreshTokenCommandHandler` shape matches the rest of the auth slice.
- `RegisterCommandHandler` and `SignInCommandHandler` extended to call `IRefreshTokenStore.IssueAsync(user.Id, familyId: null, ct)` after the access-token issuance and to return the raw token in `AuthResult.RefreshToken`. `AuthResult` record gained a `RefreshToken` field — propagated to JSON automatically.
- `AuthController` adds `POST /api/auth/refresh` accepting `RefreshTokenRequest`. The endpoint is anonymous (refresh tokens authenticate themselves via the hash lookup); no `[Authorize]` attribute.
- `Forge.Infrastructure/DependencyInjection.cs` registers `IRefreshTokenStore` as `Scoped` (matches `IAppDbContext` lifetime).

### Acceptance test (criterion 8 — ATDD evidence)

`tests/Forge.Acceptance/Auth/RefreshTokenAcceptanceTest.cs` was authored before the implementation. Header:
```
// Acceptance Test
// Traces to: BT-002 (Refresh token issuance + rotation), L2-002, L2-033
// Description: Register a user, exchange the issued refresh token once
// (succeeds), replay the consumed refresh token (401 + entire family revoked,
// so a later presentation of any sibling rotation also returns 401).
```

The test:

1. Provisions a per-test LocalDB (`Forge_Acceptance_<guid>`), pre-applies migrations directly so the DB exists before the factory boots, then spins up `WebApplicationFactory<Program>` with the connection string overridden via both `ConfigureAppConfiguration` (in-memory provider) **and** `ConfigureTestServices` (replaces the `DbContextOptions<AppDbContext>` registration). The double-override is required because `AddInfrastructure` reads the connection string eagerly at DI-build time, before the in-memory provider can take effect — the `ConfigureTestServices` swap forces the runtime DbContext onto the per-test database regardless.
2. Registers a fresh user via `POST /api/auth/register`, asserts the response carries a non-empty `refreshToken`.
3. Exchanges the refresh token once via `POST /api/auth/refresh` — asserts `200 OK` and a different (rotated) refresh token in the response.
4. Replays the original refresh token — asserts `401 Unauthorized`.
5. Presents the rotated (sibling) refresh token after the replay — asserts `401 Unauthorized` (family was revoked by the replay-detection path in `ConsumeAsync`).
6. Inspects the `RefreshTokens` table directly and asserts every row for the user has either `ConsumedAt` or `RevokedAt` populated.

Mid-implementation finding (resolved before slice closure): an initial test version used only `ConfigureAppConfiguration` to override the connection string. The runtime config object did pick up the in-memory provider, but `RefreshTokens` queried after the test posted to the API came back empty — the API was writing to the appsettings.json `Forge` database while the test was querying `Forge_Acceptance_<guid>`. Resolution: also use `ConfigureTestServices` to swap the `DbContextOptions<AppDbContext>` registration. Documented in this eval and applied to the test.

### Non-blocking observations

- The frontend's `AuthResult` model in `projects/api/src/lib/models/auth-result.model.ts` doesn't yet include `refreshToken`. Adding it — and consuming the token — is FT-014 (refreshInterceptor + authGuard) and FT-015 (Remember-me persistence). Until then, the API returns `refreshToken` and the frontend silently ignores it. No regression to L2-002 acceptance criterion 1 (the basic sign-in flow still works).
- The default refresh-token lifetime in `RefreshTokenStore` is `14 days`. BP1 noted this was unspecified by L2-033; pinning to 14 days here. Documented inline as `DefaultLifetime`. If a future requirement names a different value, change one line and add to BP1.
- `RefreshTokenStore.ConsumeAsync` does not currently check whether the **user** is soft-deleted; once BT-006 (account deletion) lands, that handler revokes refresh tokens explicitly, so reuse on a deleted account ends with the same 401 shape. Calling out so reviewers don't flag the omission.
- The acceptance test's connection-string override pattern (`ConfigureAppConfiguration` + `ConfigureTestServices`) will be reused by every subsequent BT slice that boots the API. Adding a small `ForgeAcceptanceFactory` base class is a future cleanup; not blocking BT-002.
- `MigrationsAcceptanceTest` (BT-001) doesn't need the override pattern because it doesn't boot the API — it only applies migrations directly. Behavior unchanged.
