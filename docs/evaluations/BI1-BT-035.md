# BT-035 — Readiness endpoint

## Acceptance test
`backend/tests/Forge.Acceptance/Health/ReadinessAcceptanceTest.cs` — two scenarios:
1. `GET /health/ready` against a running app with a healthy DB → 200 + `{ status: "Healthy" }`.
2. After verifying ready returns 200, drop the test DB out from under the live app and call again → 503 + `{ status: "Unhealthy" }`.

## Implementation
- `GET /health` is now liveness-only — returns `{ status: "Healthy" }` without touching the DB. (Previously `/health` ran `CanConnectAsync` which conflated liveness with readiness.)
- `GET /health/ready` runs `await _db.Database.CanConnectAsync(ct)`. Returns 200 / `{ status: "Healthy" }` when the connection succeeds; 503 / `{ status: "Unhealthy" }` when it returns `false` or throws.
- Both routes live on the existing `HealthController` to keep ops scripts pointing at the same controller; the spec only required a new method, not a new controller.
