# BT-025 — Refunds on update / delete

## How this slice was satisfied
This slice is a contract validation, not a new feature. It requires:
1. `IPointsScorer` exposes `Refund(WorkoutSession session, ct)` that appends compensating ledger rows summing to negative the awarded amount.
2. The `Update` handler calls it before re-scoring on a material change.
3. The `Delete` handler calls it before removing the session row.

All three were implemented preemptively as part of BT-010 (`UpdateSessionCommandHandler`) and BT-012 (`DeleteSessionCommandHandler`):
- `IPointsScorer.Refund` lives in `Forge.Application/Abstractions/IPointsScorer.cs`.
- The implementation is in `Forge.Infrastructure/PointsScorer.cs` — it sums all prior ledger rows for the session id and writes a single `Refund` row of `-awarded` (no-op when the sum is already 0).
- `UpdateSessionCommandHandler` calls `_scorer.Refund` before mutating fields when `Equipment`, `StartedAt`, or `DurationMinutes` changes, then `_scorer.Score` after persisting the new state.
- `DeleteSessionCommandHandler` calls `_scorer.Refund` before `Remove` + `SaveChangesAsync`, leaving the Refund row as an audit-trail tombstone tied to the deleted session id.

## Acceptance test
The BT-025 spec example (modify duration 22 → 30 → ledger has `-44 (Refund)` + `+60 (Base)`) is exercised by `UpdateSessionAcceptanceTest.Updating_duration_refunds_old_score_and_writes_new_base`. The `DeleteSessionAcceptanceTest.Deleting_a_session_returns_balance_to_pre_session_value` covers the delete leg.
