# BT-029 — Dashboard summary

## Acceptance test
`backend/tests/Forge.Acceptance/Dashboard/DashboardSummaryAcceptanceTest.cs` — one scenario:
- New user posts two same-day mid-day sessions totalling 980 kcal / 55 min in `America/New_York`. GET /api/dashboard returns:
  - `caloriesToday = 980`, `targetCalories = 1500`
  - `minutesToday = 55`, `targetMinutes = 60`
  - `currentStreak >= 1`, `currentBalance >= 0`, non-empty `tier`, `monthlyWeightGoalLb = 20`
- Total wall-clock under 1500 ms (rough perf budget — spec target is 200 ms locally; loose ceiling avoids xUnit/Kestrel cold-start flakes).

## Implementation
- `DashboardSummaryDto` aggregates: today's calories + minutes (with targets), current streak, current balance + lifetime points, tier name, next-reward-within-reach, MTD weight loss + monthly goal.
- `NextRewardDto` is a small projection used for the "next milestone" hint.
- `GetDashboardSummaryQuery() : IRequest<DashboardSummaryDto>`.
- Handler depends on `IAppDbContext` + `ICurrentUser` + `IClock`. It executes:
  1. One read for the user record (for TZ + targets).
  2. One read aggregating today's calories and minutes via `GroupBy(_ => 1)` with both sums in a single projection.
  3. One read fetching last 90 days of session start timestamps to derive `currentStreak` (anchored at `IClock.TodayInTimeZone`).
  4. One read aggregating ledger balance + lifetime in a single projection.
  5. One read for the cheapest active reward whose cost ≥ current balance.
  6. One read for the user's `WeightEntry` rows since the start of the local month, MTD weight loss = first - last.
- All reads use `.AsNoTracking()` and project to anonymous types or DTOs — no entity tracking, no second hop.
- Tier resolution walks `ScoringConstants.TierThresholds` ascending — same logic as BT-028.

## Note
The handler issues six sequential reads — fewer than a per-card read pattern. Combining further (e.g. raw SQL CTE) is a future optimisation if dashboard latency becomes a hotspot. The tier walk is in-memory over a six-element list, so no further DB cost.
