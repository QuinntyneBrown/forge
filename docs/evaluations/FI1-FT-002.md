### Mid-implementation finding (resolved before slice closure)

Initial swap moved the test selector via `data-testid="sign-in-submit"` on the `<forge-button>` host. That broke the test: Playwright clicks the host element, but DOM events propagate *up*, not down — the click never reached the inner `<button type="submit">`, the form's `(ngSubmit)` never fired, and no `POST /api/auth/sign-in` request hit the backend. Verified by adding temporary `page.on('request', ...)` listeners to the spec.

**Resolution.** Added an `@Input() testid: string | null = null` to `ButtonComponent` and bound `[attr.data-testid]="testid"` on each of the three Material button variants in the template. Consumers now pass `testid="sign-in-submit"` instead of `data-testid="sign-in-submit"`. Playwright's `getByTestId('sign-in-submit')` resolves to the inner `<button>`, the click activates the submit, the form posts, and the navigation completes.

This is a general fix that the rest of the wrapper components in FI1.0 will follow when they need to expose a stable testid hook through Material's encapsulated host (FT-003 IconButton, FT-004 Field, FT-005 Checkbox, FT-006 Switch, FT-007 Chip will all need the same `testid` input pattern — adding a note here so the consumers stay consistent).
