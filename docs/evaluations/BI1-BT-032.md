# BT-032 — Notification dispatcher

## Acceptance test
`backend/tests/Forge.Acceptance/Notifications/NotificationDispatcherAcceptanceTest.cs` — one scenario:
- FakeClock pinned at 14:00 NY, registered user has morning window 14:01-14:30 (one minute away). Calling `INotificationDispatcher.DispatchPendingAsync` once writes a `notification.morning` log line tied to the user id, captured through a test ILoggerProvider.

## Implementation
- **Abstraction:** `INotificationSender(NotificationKind { MorningWindowStarting, KitchenClosing }, userId, ct)` in `Forge.Application/Abstractions`.
- **Deferred no-op:** `Forge.Infrastructure/Deferred/LoggingNotificationSender.cs` writes `notification.morning` / `notification.kitchen` markers per `BP1 §8`. Real APNs/FCM/email transport stays out of MVP.
- **Dispatcher abstraction:** `Forge.Application/Notifications/INotificationDispatcher.DispatchPendingAsync(ct)`.
- **Dispatcher impl:** `Forge.Infrastructure/NotificationDispatcher.cs`. One read pulls users with either reminder enabled (and not soft-deleted), then for each user converts UTC `now` → user TZ and checks whether `MorningWindowStart` / `KitchenClosedStart` falls within `[now, now + 2m]` (handling midnight wrap-around). Calls the sender for each match.
- **Hosted service:** `Forge.Api/HostedServices/NotificationDispatcherHostedService.cs` runs on `PeriodicTimer(1 minute)`, creating a fresh DI scope per tick and resolving `INotificationDispatcher`. Logs `notification.dispatcher.tick.failed` on any tick error so a single failure doesn't kill the dispatcher.
- **DI:** `INotificationSender`, `INotificationDispatcher` registered in `AddInfrastructure`; `NotificationDispatcherHostedService` added in `Program.cs`.
