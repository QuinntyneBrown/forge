# FT-032 — Account deletion flow

## Acceptance test
`frontend/tests/account-deletion.spec.ts` — register, sign in, navigate
to `/profile`, click `Delete account` (reveals the confirm prompt
without destructive action), then click `Yes, delete my account`. The
test asserts the route flips to `/sign-in` and that signing in again
with the same credentials surfaces the `sign-in-error` banner. Same
SQLEXPRESS pending caveat as the recent slices.

## Implementation
### `domain` — `ProfileFormComponent`
- New signals: `confirmingDelete`, `deleting`, `deleteError`.
- Three protected methods: `requestDelete()` (reveals the confirm
  prompt), `cancelDelete()` (re-hides it), `confirmDelete()` (calls
  `IMeService.deleteMe()` and emits `(deleted)` on success).
- Two-click safety: the destructive call only runs after the user has
  seen the confirm prompt and clicked "Yes, delete my account".
- Existing `<forge-card>` body now contains a "Danger zone" section
  below the save button:
  - Title + explanatory copy.
  - Default state: a single text-style `Delete account` button.
  - Confirming state: a prompt + a destructive `Yes, delete my account`
    button (with `[loading]="deleting()"`) and a `Cancel` button.
  - `deleteError()` surfaces a friendly inline error if the API call
    fails.

### `app` — `ProfilePage`
- Now injects `AuthStateService` + `Router`. Wires
  `<forge-profile-form (deleted)="onDeleted()">`. `onDeleted()` calls
  `auth.clear()` (which also clears the persisted refresh token from
  `localStorage`) and routes to `/sign-in`.

## Library placement
- Form stays in `domain` (consumes `IMeService` from `api`). ✅
- Auth-state cleanup + router navigation stay in the `app` layer per
  the existing pattern; the form only emits an event so library
  boundaries hold.
