### Structural checks (criteria 1, 6 — Guidance adherence + SOLID/CQS shape)

- **Domain change.** `User` gains 12 profile columns:
  - `Units` (string, default `"Imperial"`) — L2-005.
  - `TimeZoneId` (string, default `"America/New_York"`) — L2-005.
  - `DailyActiveCaloriesTarget` (int, default `1500`) — L2-011.
  - `DailyWorkoutMinutesTarget` (int, default `60`) — L2-012.
  - `MonthlyWeightGoalLb` (int, default `20`) — L2-014.
  - `MorningWindowStart` / `MorningWindowEnd` (`TimeOnly`, defaults `05:00:00` / `07:30:00`) — L2-016.
  - `KitchenClosedStart` / `KitchenClosedEnd` (`TimeOnly`, defaults `20:00:00` / `06:00:00`) — L2-017.
  - `KitchenNudgeEnabled` (bool, default `true`) — L2-026.
  - `MorningReminderEnabled` (bool, default `true`) — L2-025.
  - `LeaderboardOptIn` (bool, default `false`) — L2-027.
- **Migration `AddProfileColumns`** generated via `dotnet ef migrations add`. Each column is added with its `HasDefaultValue(...)` constraint applied at the DB level, so existing user rows get the L2 defaults backfilled — no manual SQL update needed. New registrations get the same defaults via the C# property initializers.
- **`CurrentUserDto`** expanded to project every profile column. `GetCurrentUserQueryHandler` updates its EF projection accordingly. `IMeService.getMe()` on the frontend now receives the full shape; existing consumers (the dashboard greeting) keep working because TypeScript's structural typing accepts the additional fields without changes to the `CurrentUser` interface.
- **`UpdateProfileCommand`** carries the seven mutable identity-and-target fields (Email, FirstName, LastName, Units, TimeZoneId, DailyActiveCaloriesTarget, DailyWorkoutMinutesTarget). The window / boolean fields each get their own dedicated commands in BT-018 / BT-019 / BT-020 per BP1 §4.2 — splitting them this way matches the L2's intent that those are independent toggles + range controls, not part of "name + targets" identity.
- **`UpdateProfileCommandValidator`**:
  - `Email`: NotEmpty + EmailAddress + MaxLength(254).
  - `FirstName` / `LastName`: NotEmpty + MaxLength(64).
  - `Units`: must be `"Imperial"` or `"Metric"`.
  - `TimeZoneId`: NotEmpty + MaxLength(64) + a `Must(IsValidTimeZone)` rule that calls `TimeZoneInfo.FindSystemTimeZoneById` and catches `TimeZoneNotFoundException` / `InvalidTimeZoneException`.
  - `DailyActiveCaloriesTarget`: 100..10,000 (inclusive).
  - `DailyWorkoutMinutesTarget`: 0..480 (inclusive — matches L2-007's session-length bound).
- **`UpdateProfileCommandHandler`** depends on `IAppDbContext` + `ICurrentUser`. Flow:
  1. Read `userId` from `ICurrentUser` (Unauthorized if null).
  2. Look up the user; reject deleted users.
  3. If the requested email differs from the current one, query for any other user holding that email; if so, throw `EmailAlreadyRegisteredException` (already mapped to 409 by `ExceptionHandlingMiddleware` from BT-001 / register).
  4. Otherwise, update every mutable field and `SaveChangesAsync`.
- **`ProfileController`** at `api/profile`, class-level `[Authorize]`, `[HttpPut]` action returns `204 No Content`.

### Acceptance test (criterion 8 — ATDD evidence)

`tests/Forge.Acceptance/Profile/UpdateProfileAcceptanceTest.cs` was authored before the implementation. Header: `// Traces to: BT-015, L2-005, L2-014, L2-016, L2-017, L2-026, L2-027`. Two scenarios:

1. **Defaults + update round-trip.** Register a fresh user. `GET /api/me` before any update — asserts every L2-defaulted field returns the documented value (`Imperial`, `1500`, `60`, `20`, `05:00:00`, `07:30:00`, `20:00:00`, `06:00:00`, `KitchenNudgeEnabled=true`, `MorningReminderEnabled=true`, `LeaderboardOptIn=false`). PUT `/api/profile` with a new email + new identity + Metric / America/Toronto / 1800 / 75. `GET /api/me` after — asserts every change persisted.
2. **Email collision.** Register two users. The second user PUTs their email to the first user's email. Asserts `409 Conflict`.

The first scenario doubles as the "defaults from migration land correctly on a fresh registration" check — important because the migration's `HasDefaultValue` applies at the DB level, and a registration that doesn't set the value relies on EF reading the default back from the database after `SaveChangesAsync`. Verified end to end.
