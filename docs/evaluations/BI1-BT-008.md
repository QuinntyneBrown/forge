### Structural checks (criteria 1, 6 — Guidance adherence + SOLID/CQS shape)

- **`IAuditLogger`** lives in `Forge.Application/Abstractions/`. Single async method `WriteAsync(string @event, Guid? userId, object? payload, CancellationToken)`. Intentionally minimal — handlers don't reach for IP / user-agent themselves; the implementation derives those from the ambient HTTP context, keeping the Application layer free of HTTP concerns.
- **`AuditLogger`** in `Forge.Infrastructure/`:
  - Reads IP and User-Agent from `IHttpContextAccessor.HttpContext`. Null-safe — when no request is in flight (e.g., a future hosted-service slice), both fields are stored as null.
  - Stamps `OccurredAt` from `IClock.UtcNow` (BT-006a seam).
  - Serializes `payload` to JSON via `System.Text.Json` and runs a recursive `Redact` over the resulting `JsonNode` tree, replacing the value of any key in `RedactedKeys` (`password`, `newPassword`, `currentPassword`, `accessToken`, `refreshToken`, `passwordResetToken`, `token`, `authorization`) with the literal `"[REDACTED]"`. The redaction walks both `JsonObject` and `JsonArray` children so nested DTOs are scrubbed too.
  - Persists via `IAppDbContext` (no repository / unit-of-work). Registered as `Scoped` in `Forge.Infrastructure/DependencyInjection.cs` (matches the `IAppDbContext` lifetime so audit writes participate in the same scope as their handler's other writes — the audit row goes in the same `SaveChangesAsync` transaction).
- **Handler integration** (4 wired in this slice; BT-005 and BT-006 will wire their own when they land):
  - `RegisterCommandHandler` writes `register.success` with `{ Email }` payload.
  - `SignInCommandHandler` writes `sign-in.failure` (with `{ Email }`) before throwing on bad credentials, and `sign-in.success` (with `{ Email }`) after issuing tokens.
  - `SignOutCommandHandler` writes `sign-out` with the current user id from `ICurrentUser` (now injected) and a null payload.
  - `RefreshTokenCommandHandler` writes `token.refresh.failure` on the unhappy paths and `token.refresh.success` after rotation. Failure rows audit with the `consumed.UserId` when known, otherwise `null` (when the token didn't even resolve).

### Acceptance test (criterion 8 — ATDD evidence)

`tests/Forge.Acceptance/Auth/AuditLogAcceptanceTest.cs` was authored before the handler integrations. Header carries `// Traces to: BT-008, L2-035, L2-051`. The test:

1. Per-test LocalDB + WebApplicationFactory using the canonical override pattern.
2. Registers a user with a distinctive password (`"ZebraQuokka!9z!2026"`) — chosen to be unique so a substring search has no false positives.
3. Sends a sign-in with the wrong password — asserts `401`.
4. Sends a sign-in with the correct password — asserts `200`.
5. Reads `AuditLogs` directly from a fresh `AppDbContext` and asserts:
   - `register.success` row exists.
   - `sign-in.failure` row exists.
   - `sign-in.success` row exists.
   - **No row's `PayloadJson` contains the distinctive password substring** (L2-051 redaction).

The test exercises the redaction by sending the password through the handler path that audit-logs `{ Email }` — the password isn't passed to the audit writer, but the test asserts the broader invariant that distinctive password strings never reach any audit row regardless of which payload shape a handler chooses.

### Non-blocking observations

- The `AuditLogger.Redact` recursive walk does not strip the `Email` field. That's intentional — email is part of the audit entry's value (knowing *who* attempted sign-in is the whole point) and is not a secret per L2-051. If a future slice introduces a payload that contains an email solely as PII (not as the actor identifier), the redaction set can be extended.
- Failed sign-ins for unknown emails record `userId = null`. That's correct behavior for L2-035 — we still capture the attempt without inventing a user-id.
- The audit rows pile up in the same transaction as the operation they describe; if a handler throws *after* the audit write but before its primary `SaveChangesAsync`, the audit row may still be persisted because the audit writer calls its own `SaveChangesAsync`. For the sign-in failure path (the only handler that throws after auditing), this is fine — we *want* the failure recorded even if downstream code throws. Documented here so a future reviewer doesn't expect strict transactional unity between the audit row and the handler's primary write.
- Future slices that add hosted services / background workers will need to handle `HttpContextAccessor.HttpContext == null` — the implementation already does this gracefully (IP/UA stored as null). Any future call site can audit without scaffolding.
