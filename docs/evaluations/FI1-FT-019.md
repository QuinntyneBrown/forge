# FT-019 — DailyRingCardComponent

## Acceptance test
Per the FT-019 entry, the dedicated assertion lives in `dashboard.spec.ts` (FT-024). FT-019 is a domain card whose visual + data wiring is exercised when the dashboard page renders.

## Implementation
- `frontend/projects/domain/src/lib/daily-ring-card/daily-ring-card.component.{ts,html,scss}`.
- Injects `IDashboardService` via the `DASHBOARD_SERVICE` token (constructor injection per existing `health-badge` pattern).
- State held in signals (Angular 21 zoneless): `summary`, `errored`, plus computed `caloriesToday`, `targetCalories`, `minutesToday`, `targetMinutes` so the template stays declarative.
- Composes `<forge-card>` + `<forge-progress-ring>` (both from the `components` library) with a centered numeric overlay (`{caloriesToday()} / {targetCalories()} kcal`). A sibling tile shows minutes-today vs `targetMinutes`.
- BEM class names throughout (`.daily-ring-card`, `.daily-ring-card__inner`, etc.).
- `data-testid` attributes on the card, ring, value, and minutes tile so FT-024 can assert without coupling to inner Material classnames.
- Public-API export added in `projects/domain/src/public-api.ts`.

## Library placement
- Lives in `domain` (consumes the `api` layer via `IDashboardService`). Imports `components` (`CardComponent`, `ProgressRingComponent`) and `api` only — no `app` imports. ✅

## Build verification
- `npx ng build forge` succeeds with the new component included via path-mapped library compilation. The `ng build domain` standalone ng-packagr build fails with a pre-existing `Cannot destructure property 'pos' of 'file.referencedFiles[index]'` error that reproduces on `main` even before this slice — independent of FT-019. Filed informally as a follow-up; not blocking since the app build path is the one used in dev / test / Playwright.
