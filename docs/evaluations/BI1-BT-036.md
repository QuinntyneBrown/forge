# BT-036 — CI dependency vulnerability scan

## Acceptance test
The spec scenario ("intentionally pin a known-vulnerable dep → CI fails; remove → CI passes") is validated by GitHub Actions itself once the workflow runs against a draft PR. There is no equivalent in-process xUnit acceptance test; the contract is the workflow file.

The local backend test suite (53/53) still runs as the same `dotnet test` step the CI job invokes, which exercises the build/test path the workflow depends on.

## Implementation
- `.github/workflows/ci.yml` runs on every push to `main` and every PR targeting `main`.
- **backend job** (`windows-latest`):
  1. `dotnet restore`
  2. `dotnet build --configuration Release`
  3. `dotnet test --configuration Release`
  4. `dotnet list package --vulnerable --include-transitive` — output is captured; the step fails if the text contains `high` or `critical`.
- **frontend job** (`ubuntu-latest`):
  1. `npm ci`
  2. `npm run build --if-present` (no-op if no build script defined)
  3. `npm audit --omit=dev --audit-level=high` — fails on any high/critical advisory in production deps.

## Note
The workflow file is the contract; the negative scenario is validated by the GitHub Actions runtime when a vulnerable dep is intentionally added in a draft PR. There is no value in mocking that locally — the goal is the actual cloud check.
