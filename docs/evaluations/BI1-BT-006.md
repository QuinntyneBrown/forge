### Structural checks (criteria 1, 6 — Guidance adherence + SOLID/CQS shape)

- **Domain change.** `User` entity gains `IsDeleted` (bool, default false) and `DeletedAt` (nullable `DateTimeOffset`). Migration `AddSoftDeleteToUsers` was generated via `dotnet ef migrations add` and applies cleanly on top of the existing `Initial_AuthAndSessions` migration (per BT-001's switch from `EnsureCreated` to `MigrateAsync`).
- **`DeleteAccountCommand : IRequest`** (no payload — the user id comes from the JWT via `ICurrentUser`). Empty `DeleteAccountCommandValidator` exists so the FluentValidation pipeline behavior runs uniformly with every other command in the slice.
- **`DeleteAccountCommandHandler`** dependencies: `IAppDbContext`, `ICurrentUser`, `IRefreshTokenStore`, `IClock`, `IAuditLogger`. Flow:
  1. Read `_currentUser.UserId`; if null, throw `UnauthorizedAccessException`.
  2. Look up the user; if missing or already `IsDeleted`, throw `UnauthorizedAccessException` (no second-delete enumeration channel).
  3. Soft-delete: `IsDeleted = true`, `DeletedAt = _clock.UtcNow`.
  4. Anonymize PII columns: `Email = $"deleted+{user.Id:N}@forgefit.local"` (uses the user's id so the unique-on-email index stays satisfied), `FirstName = "Deleted"`, `LastName = "User"`. `PasswordHash` is also blanked so the sentinel email can never sign in even if guessed.
  5. `SaveChangesAsync` persists the User row.
  6. Call `IRefreshTokenStore.RevokeAllForUserAsync(user.Id, ct)` (the BT-005 helper) so every active refresh-token family is invalidated (L2-033).
  7. Audit `account.delete`.
- **`MeController`** (new) at `api/me`. `[Authorize]` at the controller level. `DELETE` action sends `DeleteAccountCommand` and returns `204 No Content`.
- All persistence flows through `IAppDbContext` (no repository / UoW). `ICurrentUser` keeps the handler decoupled from `IHttpContextAccessor`.

### Non-blocking observations

- The L2-006 ac 1 mentions "confirm twice" before deletion. That's a UI guard belonging to FT-032 (the frontend account-deletion flow). The backend endpoint is single-action by design — the second confirm is handled in the consumer.
- `L2-050` says PII fields are anonymized "within 24 hours" via a nightly cleanup job. This implementation anonymizes inline at delete time — a stricter guarantee than the L2 requires, and it eliminates the gap between deletion and cleanup. No background job is needed; the L2 ac 1 query ("when the user record is inspected after the next nightly cleanup job, `Email`, `FirstName`, and `LastName` are null or hashed sentinel values") is satisfied by the immediate write.
- The user's `WorkoutSessions` rows still reference the `UserId` after deletion. They retain non-PII payload (equipment, duration, calories, etc.). Whether to also delete or anonymize those rows is a future decision (L2-050 is silent on user-owned content). For the MVP, leaving them in place avoids cascading deletes and preserves audit trails.
- Audit rows referencing the deleted user keep their `UserId` column populated for trace integrity (L2-050 ac 2 explicitly preserves this). The audit logger's redaction (BT-008) ensures the audit row's `PayloadJson` has never contained the email or PII; only the user id remains.
- A future BT-005-style flow (password reset for a deleted user) is harmless: the `RequestPasswordResetCommandHandler` looks up by the original email, which now no longer matches a user, so the request silently no-ops (still returns 202 — no enumeration). Calling out so reviewers don't flag the deleted-user-then-reset path as a bug.
