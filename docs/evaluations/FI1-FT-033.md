# FT-033 — SyncErrorPanelComponent + HealthKitService + ErrorPage

## Acceptance test
`frontend/tests/error-state.spec.ts` opens `/error?traceId=abc123def456`,
asserts the panel renders the trace id and the diagnostics, and that
the "Go to dashboard" CTA fires (anonymous user gets redirected to
`/sign-in` via `authGuard` — the redirect itself confirms the click
fired correctly). Same SQLEXPRESS pending caveat for the actual
backend `/health` call; the spec asserts the badge text matches
`Healthy|Unhealthy|Loading` so it works in either state.

## Implementation
### `api` library
- `healthkit.service.contract.ts` — `IHealthKitService.ingest(sample)` + `HEALTHKIT_SERVICE` token; `HealthKitSample { sampleType, value, unit, recordedAt }` matching BT-031.
- `healthkit.service.ts` — calls `POST /api/healthkit/ingest`.
- Public-API exports added.

### `domain` library
- `sync-error-panel/sync-error-panel.component.{ts,html,scss}`:
  - Composes `<forge-card>`, `<forge-error-banner>`, `<forge-badge>`, `<forge-button>`.
  - Reads live status from `IHealthService.getStatus()`; reflects `Healthy`/`Unhealthy`/`Unreachable` via badge variant.
  - Surfaces `[traceId]` input as `<code>` block so support can copy it.
  - "HealthKit ingest" badge stays "Deferred" — the BT-031 stub doesn't expose a status query, and the panel only needs to communicate "this transport is a no-op for MVP" to the user.
  - "Go to dashboard" CTA navigates to `/dashboard`.
- Public-API export added.

### `app`
- `pages/error/error.page.{ts,html,scss}` reads `?traceId` from `ActivatedRoute` and feeds it into the panel.
- Route `/error` (no `authGuard` — error routes must be reachable when auth fails).

### DI
- `{ provide: HEALTHKIT_SERVICE, useClass: HealthKitService }` registered in `app.config.ts`.

## Library placement
- Service in `api`. ✅
- Panel in `domain`. ✅
- Page in `app`.
