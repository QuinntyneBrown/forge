The following blocking findings were found:

### Finding 1 — SOLID / Clean Architecture / CQS not captured as a requirement (R2 criterion 5: Guidance coverage)

The Implementation Guidance "General" and "Backend" sections require S.O.L.I.D. principles, Clean Architecture (Api / Application / Domain / Infrastructure), Command/Query Separation via MediatR, `IAppDbContext`-driven handlers, and one-type-per-file. R2 criterion 5 names "SOLID" explicitly. None of L2-001..L2-051 captures this; it is treated as an implicit code-design constraint without a behavioral acceptance criterion that BI1/BT1 evaluators can trace tests to.

**Fix:** add a new L1 ("Architectural integrity") and an L2 with grep- / structure-check-style acceptance criteria covering Clean Architecture project split, MediatR-only command/query flow, `IAppDbContext` dependency, no repository/unit-of-work classes, one type per `.cs` file, and one-type-per-file Angular components.

### Finding 2 — Output encoding / XSS protection missing (R2 criterion 8: OWASP)

R2 criterion 8 calls out "output encoding" as an OWASP-relevant L2 the spec must include. The current L2 set covers input validation (L2-039, L2-040), authentication (L2-031..L2-036), access control (L2-037, L2-038), secrets handling (L2-051), and audit logging (L2-035) — but contains no L2 that requires output encoding / XSS prevention on the rendered Angular surfaces or in API responses that echo user input.

**Fix:** add an L2 under a new "application security baseline" L1 stating that user-supplied content shall be HTML-encoded by Angular's default interpolation, no `[innerHTML]` shall be used with unsanitized user input, and the `Content-Security-Policy` response header shall forbid inline scripts and unsafe-eval.

### Finding 3 — Dependency / supply-chain hygiene missing (R2 criterion 8: OWASP)

OWASP Top 10 includes "Vulnerable and Outdated Components" (A06:2021). The current L2 set never names dependency scanning, lockfile review, or a CVE policy. With Argon2/PBKDF2/JWT/EF Core/Angular libraries present, an unscanned dependency tree is a real risk.

**Fix:** add an L2 requiring `npm audit` / `dotnet list package --vulnerable` (or equivalent) to run in CI with build-failing thresholds, and lockfiles to be committed.

### Non-blocking observations

- Several UI L2s (L2-001 registration, L2-002 sign-in, L2-007 log session, L2-024 error-state) do not name breakpoint behavior in their acceptance criteria. R2 criterion 7 says "where relevant"; the cross-cutting L2-030 ("layout shall reflow … without horizontal scrolling, clipped content, or overlapping elements" at XS/SM/MD/LG/XL) covers them and L2-046 covers touch-target size. Treating L2-030 as the catch-all is defensible and not blocking.
- L2-022 (tier progression) names one threshold by example (`Forged Iron` at 5,000 lifetime points) but does not enumerate all five tier thresholds. The criterion is testable as written ("deterministic"), and R1 was scoped to MVP — leaving the full table for the implementation slice that owns the rewards economy.
- The traceability table in `L2.md` lists `L2-005` twice for the profile screen. Cosmetic only; not a blocker.
