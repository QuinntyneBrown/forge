# FT-029 — WorkoutDetailPage + WorkoutPointsBreakdownComponent

## Acceptance test
`frontend/tests/workout-delete-refund.spec.ts` seeds a 22-min treadmill
session at mid-day NY (so only the +44 Base ledger row writes), signs
in, asserts the tier card reads "44 pts available", opens
`/workouts/:id`, clicks Delete, routes back to the dashboard, asserts
the tier card now reads "0 pts available". Same SQLEXPRESS pending
caveat — the spec is in tree but live execution needs the local DB
instance running.

## Implementation
### `domain` library
- Extended `WorkoutDetailFormComponent` with edit-mode support:
  - `[mode]: 'create' | 'edit'`, `[sessionId]`, `[session]` inputs.
  - `applySession()` patches the form when `[session]` flips from `null`.
  - On submit in `edit` mode the form calls `ISessionsService.update` and emits `(saved)`; create mode is unchanged.
  - Submit button label switches between "Save session" and "Save changes" via a computed.
- Added `WorkoutPointsBreakdownComponent`:
  - `[session]: Session | null` input. Recomputes the breakdown rows on input change.
  - Renders the Base row of `BasePointsPerMinute × DurationMinutes` (mirrors `ScoringConstants.BasePointsPerMinute = 2` from the backend).
  - A footnote calls out that morning bonus and streak multiplier are computed by the backend and reflected in the dashboard balance — keeping the breakdown honest without needing a new endpoint.
- Public-API exports added.

### `app`
- `pages/workout-detail/workout-detail.page.{ts,html,scss}`:
  - Reads `:id` from `ActivatedRoute`, fetches the session via `ISessionsService.getById`, feeds it into both the form (edit mode) and the breakdown.
  - "Duplicate" / "Delete session" CTAs wire to `ISessionsService.duplicate` / `delete`. Save / duplicate route to `/workouts`; delete routes to `/dashboard`.
  - `[disabled]` and `busy` signal block double-clicks.
- `app.routes.ts` registers `/workouts/:id` behind `authGuard` (after `/workouts/new` so route matching prefers the static segment).

## Library placement
- Components in `domain` (consume `api`). ✅
- Page in `app`.

## Notes on points-breakdown scope
The slice spec said "renders the points ledger rows for the session
(Base, Morning bonus, Streak multiplier)". The current backend
`SessionDto` does not include ledger rows and BI1 is closed. To avoid
re-opening BI1 for a single read endpoint, the breakdown shows the
deterministic Base row computed client-side and a footnote pointing the
user to the dashboard balance for the bonus rows. Adding a
`GET /api/sessions/{id}/ledger` endpoint is a clean follow-up if/when
the design requires per-session bonus visibility.
