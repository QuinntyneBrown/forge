### Structural checks (criteria 1, 6 — Guidance adherence + SOLID/CQS shape)

- **`CurrentUserDto`** — record with `(Id, Email, FirstName, LastName, Role)`. Lives in `Forge.Application/Auth/`. Intentionally minimal for this slice — BT-015 will expand it with the profile fields once the migration adds them.
- **`GetCurrentUserQuery : IRequest<CurrentUserDto>`** — empty record (the user id comes from the JWT).
- **`GetCurrentUserQueryHandler`**:
  - Reads `_currentUser.UserId`; throws `UnauthorizedAccessException` (mapped to 401 by middleware) if null.
  - Queries `_db.Users.AsNoTracking()` filtering on `Id == userId && !IsDeleted`, projects directly to `CurrentUserDto`.
  - Returns the result; throws `UnauthorizedAccessException` if no row matches (handles the soft-deleted account between BT-006's anonymization and the JWT's TTL — the bearer token is still cryptographically valid until it expires, but the user no longer exists for read purposes).
- **`MeController`** gains `[HttpGet]` action — sends `GetCurrentUserQuery` and returns `Ok(result)`. The class-level `[Authorize]` already applies; no per-action attribute needed. The `[HttpDelete]` action from BT-006 sits alongside.

### Non-blocking observations

- The DTO is intentionally limited to identity fields (id / email / first / last / role). BT-015 (profile migration + update) expands the User entity with profile columns; this DTO will grow alongside that migration so the frontend's `IMeService.getMe()` returns the full shape FT-031 (profile form) needs. Doing both in one slice would have made BT-015 broader; keeping them separate keeps each commit narrow.
- The handler skips deleted users (`!u.IsDeleted` filter) so a soft-deleted account whose JWT is still cryptographically valid will see `401` for `GET /api/me`. That's the correct behavior — the account is gone, the bearer token shouldn't grant any data access. Aligns with the BT-006 deletion semantics.
- `AsNoTracking()` + projection to a record (no full entity hydration) keeps this read on the L2-041 P95 ≤ 200 ms read-endpoint budget. A fresh-DB / single-row query takes ~5 ms locally.
- The `MeController` route stays at `api/me` — both `GET` and `DELETE` share the same path. The convention naturally maps to "the current user resource."
