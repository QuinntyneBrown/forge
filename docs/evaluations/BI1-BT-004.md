### Structural checks (criteria 1, 6 — Guidance adherence + SOLID/CQS shape)

- **`IPasswordResetEmailSender`** in `Forge.Application/Abstractions/`. Single async method `SendAsync(email, rawToken, ct)`. The Application layer doesn't know transport details (SMTP, SES, SendGrid); the contract is simply "emit a reset link to this address."
- **`LoggingPasswordResetEmailSender`** in `Forge.Infrastructure/Deferred/` per BP1 §8. Logs `password-reset.email.deferred email={Email} token={Token}` via `Microsoft.Extensions.Logging`. Class XML doc-comment names it as the deferred-integration no-op so a future grep finds it. Registered Scoped in DI.
- **`RequestPasswordResetCommand : IRequest`** (no result — fire-and-forget). Colocated `RequestPasswordResetCommandValidator` (FluentValidation: `NotEmpty`, `EmailAddress`, `MaximumLength(254)`). Handler:
  - Looks up the user by normalized email. If not found → returns immediately (still 202 to caller — no enumeration).
  - If found → generates a 32-byte cryptographically random raw token (base64-url, no padding), persists only its SHA-256 hex hash in `PasswordResetTokens` with `IssuedAt = _clock.UtcNow` (BT-006a seam) and `ExpiresAt = IssuedAt + 30 minutes` (constant `TokenLifetime` exposed as `public static readonly` for test access).
  - Calls `_sender.SendAsync(normalizedEmail, rawToken, ct)`. The raw token leaves the handler exactly once — at send time.
- **`AuthController`** adds `[HttpPost("password-reset/request")]` (anonymous). Returns `Accepted()` (202) regardless of whether the user existed.

### Acceptance test (criterion 8 — ATDD evidence)

`tests/Forge.Acceptance/Auth/PasswordResetRequestAcceptanceTest.cs` was authored before the implementation. Header: `// Traces to: BT-004, L2-004`. Two scenarios:

1. **Unknown email**. POST `/api/auth/password-reset/request` for `unknown+<guid>@forgefit.app` → asserts 202 Accepted and `PasswordResetTokens.Count == 0` in the per-test DB.
2. **Known email**. Register a user, then POST the reset request → asserts 202, exactly one `PasswordResetTokens` row exists, its `TokenHash` is non-empty, `ConsumedAt` is null, and `ExpiresAt - IssuedAt` lies in `(0, 30 minutes]`.

Both scenarios share the canonical `WebApplicationFactory<Program>` override pattern (per-test LocalDB + DbContext swap). The test deliberately doesn't decode the raw token — the redaction-by-design store-only-hash invariant is exercised by inspecting the persisted row's `TokenHash`.

### Non-blocking observations

- BP1 specified an audit row `password-reset.success` for the *confirm* leg. The *request* leg currently doesn't write an audit row — by design, since the unknown-email branch returns silently and writing different audit shapes for unknown vs known would itself be an enumeration channel. BT-005 (confirm) will add `password-reset.success` and `password-reset.failure` audit events.
- The 30-minute TTL is fixed in code as a `public static readonly TimeSpan` on the handler. If a future requirement makes it configurable, swap to `IOptions<...>` — one change to the constructor.
- The handler doesn't enforce a one-pending-token-per-user invariant. A user who repeatedly requests reset will end up with multiple unconsumed token rows; only the most recent is shown by the email log. BT-005 will validate the *presented* token, so multiple pending tokens don't compromise security — they just clutter the table. A future cleanup slice can add a sweep job; not blocking BT-004.
- Hashing the token with SHA-256 + base64-url encoding is the same pattern as `RefreshTokenStore` (BT-002). Keeping the convention identical means a future audit can verify "no raw secrets are persisted" with a single grep across `Forge.Infrastructure`.
