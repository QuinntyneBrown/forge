# BT-033 — Security headers middleware

## Acceptance test
`backend/tests/Forge.Acceptance/Security/SecurityHeadersAcceptanceTest.cs` — one scenario:
- `GET /health` returns 200 plus Content-Security-Policy (containing `default-src 'self'`, `script-src 'self'`, `frame-ancestors 'none'`, `object-src 'none'`), `X-Content-Type-Options: nosniff`, `Referrer-Policy: no-referrer`.

## Implementation
- `Forge.Api/Middleware/SecurityHeadersMiddleware.cs` registers headers on `Response.OnStarting` so they're emitted regardless of how the response is built (including responses written by `ExceptionHandlingMiddleware`).
- `Content-Security-Policy: default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'; object-src 'none'; frame-ancestors 'none'`
- `X-Content-Type-Options: nosniff`
- `Referrer-Policy: no-referrer`
- `Strict-Transport-Security: max-age=31536000; includeSubDomains` — gated to non-`Development` environments so local HTTP loops aren't pinned to HTTPS by browsers.
- `Program.cs` registers the middleware as the first pipeline step, before `ExceptionHandlingMiddleware`, so error responses also carry the headers.
