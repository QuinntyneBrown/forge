### Implementation

- **`CurrentUser` model expansion.** Now carries every BT-015 column (units, timeZoneId, three int targets, four window times as ISO `HH:MM:SS` strings, three booleans). The dashboard's existing `currentUser()` signal continues to work — TypeScript adds the fields without runtime changes.
- **`ProfileService.updateProfile(req)`** — PUTs to `/api/profile`. Bearer + refresh interceptors handle auth attachment and 401 retries.
- **`ProfileFormComponent`**:
  - Reactive form with seven controls: `firstName` / `lastName` / `email` (length + email validators), `units` (string-literal `'Imperial' | 'Metric'`, required), `timeZoneId` (string, required, max 64), `dailyActiveCaloriesTarget` (100..10,000), `dailyWorkoutMinutesTarget` (0..480) — mirrors backend `UpdateProfileCommandValidator` so the user gets immediate client-side feedback.
  - `ngOnInit` calls `meApi.getMe()` and populates the form via `setValue`. The `@Input() user` allows a parent page to pre-supply the user (avoiding a duplicate fetch); the host `ProfilePage` doesn't pre-supply, so the form's own `getMe()` runs.
  - On submit, calls `profile.updateProfile(form.getRawValue())`. On 204 sets a `saved` signal that surfaces "Saved." copy. On error reads `err.error.title` from the `ProblemDetails` body.
  - State held in **signals** (`errorMessage`, `saved`, `submitting`) — required for change detection in zoneless mode.
- **`ProfilePage`** wraps the form in `<forge-app-shell>` so the bottom-nav / nav-rail / top app bar render alongside it, consistent with the dashboard layout.

### Mid-implementation finding (resolved before slice closure)

**Full-page navigation drops the in-memory session** unless Remember me is checked. The first test draft signed in without Remember me, navigated to `/profile` via `page.goto('/profile')`, and the auth guard bounced the test back to `/sign-in`. **Resolution.** Test now clicks the Remember-me checkbox before signing in, so `localStorage` carries the refresh token and `provideAppInitializer(tryHydrate)` rehydrates the session before the auth guard runs.

**Two consecutive `page.goto`s consume too many refresh-token rotations.** A second-stage flake: after submitting the form, navigating `/dashboard` → `/profile` runs `tryHydrate` twice in a row. BT-002's replay-detection sees the second-stage rotated token and revokes the family if there's a race. **Resolution.** Replaced the navigation hop with a single `page.reload()` — same persistence guarantee, one rehydration cycle, no race.
