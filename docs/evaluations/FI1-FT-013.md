### Implementation

- **`IAuthService.requestPasswordReset(email)`** returns `Observable<void>` and POSTs `{ email }` to `/api/auth/password-reset/request` (the BT-004 endpoint).
- **`PasswordResetRequestFormComponent`** (`<forge-password-reset-request-form>`):
  - Reactive form with one `email` control (`required`, `email`, `maxLength(254)` — mirrors backend `RequestPasswordResetCommandValidator`).
  - On submit, calls `auth.requestPasswordReset(email)` and flips a `submitted` signal regardless of `next` / `error`. The error path also flips submitted because the request leg is designed to never reveal account existence — even network errors land on the same confirmation copy (L2-004 ac 1 — no enumeration).
  - The `submitted` signal swaps the template from the form to a confirmation paragraph carrying the testid `password-reset-request-confirmation` and the copy "If an account exists for that email, a reset link is on its way. Check your inbox." Identical wording for both code paths.
- **`PasswordResetPage`** wraps the form in a hero/card layout that mirrors `SignInPage` and `SignUpPage` for visual consistency. 2-column at ≥768px, stacked at <768px. The page is anonymous (no auth guard) — accessible from the sign-in form's "Forgot password?" link (already present in MF1's mock; the actual `<a routerLink>` lives outside this slice's scope and lands in a follow-up).

### Acceptance test (criterion 8 — ATDD evidence)

`tests/password-reset-request.spec.ts` was authored before the implementation. Header: `// Traces to: L2-004 (request leg)`. The test:

1. Pre-registers one account via the live backend so we can drive the known-email branch against a real DB row.
2. Loops over `[knownEmail, unknownEmail]` and for each:
   - Navigates to `/password-reset`.
   - Fills the email, clicks submit.
   - Waits for the `POST /api/auth/password-reset/request` response and asserts `202 Accepted`.
   - Asserts `getByTestId('password-reset-request-confirmation')` becomes visible.

Both branches share the exact same assertion set — the test mechanically enforces "no account enumeration via response shape" because any attempt to differentiate would have to add a new testid or check a different element, immediately breaking the parity.

### Non-blocking observations

- The confirm leg of L2-004 (FT-013-confirm) is **deferred**. It needs:
  - Backend BT-005 (`ConfirmPasswordResetCommand` + `/api/auth/password-reset/confirm` endpoint).
  - Frontend `PasswordResetConfirmFormComponent` (in `domain`) + a token-aware route `path: 'password-reset/confirm'` (or a query-param toggle on the existing page per FP1 §5).
  - A spec extension covering token-validation + new-password submission + sign-in with the new password.
- The "Send reset link" button uses `[loading]="submitting"` to show the in-button spinner from FT-002. After submission, the entire form is replaced by the confirmation paragraph — the Material spinner only flashes briefly on the network round-trip. Acceptable UX for an MVP.
- The form deliberately doesn't surface an error message even when the backend returns 5xx — the request leg's threat model treats every response as "we're handling it; check your email." A real production deployment may want to surface a generic "Try again later" toast on hard failures; that's a polish item the FI1.6 (empty + error states) phase can pick up.
- `forge-button` carries `testid="password-reset-request-submit"` (forwarded as `data-testid` on the inner `<button mat-flat-button>` per the FT-002 pattern).
