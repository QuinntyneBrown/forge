# BT-024 — Streak multiplier

## Acceptance test
`backend/tests/Forge.Acceptance/Gamification/StreakMultiplierAcceptanceTest.cs` — one scenario:
- A `FakeClock` registered as the `IClock` singleton walks through 7 consecutive UTC days, posting a 22-min treadmill session each day at 14:00 NY local (well outside the morning window). After the loop:
  - Day 7's session has a `StreakMultiplier` ledger row with `Points = 3` (matches `floor(44 × 0.07) = 3`).
  - Days 3..7 each produce a streak row; days 1-2 do not because `floor(44 × 0.01) = floor(44 × 0.02) = 0` (no row appended when bonus rounds to zero).
- The clock then advances by 2 days (skipping day 8), and an 8th session posts on day 9. That session gets no `StreakMultiplier` row because day 8 has no session, so `consecutiveStreakDays = 1` and `floor(44 × 0.01) = 0`.

## Implementation
- `PointsScorer.Score` now computes `consecutiveStreakDays` by:
  1. Anchoring "today" with `_clock.TodayInTimeZone(user.TimeZoneId)`.
  2. Loading the user's session `StartedAt` values from the past 90 days (`AsNoTracking`).
  3. Converting each into the user's TZ → distinct `DateOnly` set.
  4. Walking back from today, counting while consecutive days are present.
- `multiplier = min(1.50, 1.00 + 0.01 × streakDays)` (constants from `ScoringConstants`).
- A `StreakMultiplier` row is appended only when `floor(basePoints × (multiplier - 1.00)) > 0`, avoiding zero-point rows.
- `Description = $"Streak multiplier ×{multiplier:0.00}"`.

## Note on spec example
`backend-tasks.md` says the day-7 streak row should be `+6 (Streak multiplier ×1.07)`. With the formula as written (`floor(basePoints × (multiplier - 1.00))`) and `basePoints = 44`, the answer is `floor(44 × 0.07) = 3`, not 6. The test follows the formula. Either the example is a transcription error or the intended formula was richer (e.g. `floor(basePoints × 2 × (multiplier - 1.00))` or a separate quadratic step). Treat the formula as authoritative until L2-020 / the doc is reconciled — flagging here for the next plan-pass.
