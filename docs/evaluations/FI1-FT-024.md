# FT-024 — DashboardPage rewrite + spec

## Acceptance test
`frontend/tests/dashboard.spec.ts` — one scenario:
- Register a fresh user via the API, sign in via the UI, land on `/dashboard`.
- All five cards (`daily-ring-card`, `streak-card`, `weight-progress-card`, `tier-card`, `leaderboard-card`) render.
- For a brand-new user with default profile values:
  - Daily ring shows "60 min today" (default workout target).
  - Streak shows 0 days, multiplier badge `×1.00`.
  - Weight goal shows "20 lb / month", MTD `0.0 lb so far`.
  - Tier name "Iron", balance "0 pts available".
  - Leaderboard contains the caller themselves (always-include rule from BT-030).

## Implementation
- `projects/forge/src/app/pages/dashboard/dashboard.page.{ts,html,scss}` rewritten:
  - Imports `DailyRingCardComponent`, `StreakCardComponent`, `WeightProgressCardComponent`, `TierCardComponent`, `LeaderboardCardComponent` from the `domain` library.
  - `<forge-app-shell>` wraps the existing greeting + sign-out header and a 2-column responsive grid containing the five cards. The daily ring and leaderboard span both columns at desktop width via `dashboard__cell--span-2`.
  - HealthBadge composition removed (it moves to `error-state.html` per FT-033).
- `frontend/tests/pom/dashboard.page.ts` — POM helper extended with locators for every new test-id surfaced by the cards.
- `frontend/tests/sign-in.spec.ts` — relaxed: no longer asserts the health badge on the dashboard (still asserts greeting email + role). Documented the FT-024 / FT-033 split inline.
- `CardComponent` (in `components`) gained a `[testid]` input forwarded to `<mat-card>` via `[attr.data-testid]` so each card surfaces a stable hook for Playwright. Followed the same pattern used by `ButtonComponent`, `CheckboxComponent`, `SwitchComponent`, etc.

## Library placement
- All five card components stay in `domain`; the page in `app` composes them. ✅
