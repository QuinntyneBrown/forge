# BT-034 — Structured logging + secret redaction filter

## Acceptance test
`backend/tests/Forge.Acceptance/Logging/StructuredLoggingAcceptanceTest.cs` — three scenarios:
1. Register with a distinctive password → no captured log entry's `Formatted` text or structured property values contains the password string.
2. Hit `/health` → exactly one `request.handled` entry exists with `Method`, `Path`, `Status`, `TraceId`, `UserId` properties.
3. Synthetic probe via `ILoggerFactory` logs sensitive structured params (`password`, `accessToken`, `refreshToken`, `Authorization`) → values are masked to `***` in both the formatted message and the structured state, while non-sensitive params (`Email`) remain intact.

## Implementation
- **JSON console formatter** — `builder.Logging.AddJsonConsole(...)` configures stdout to emit one JSON object per log line. Verified by inspecting actual stdout in the test run output (every line is parseable JSON with `EventId`, `LogLevel`, `Category`, `Message`, `State`, `Scopes`).
- **Redacting wrapper** — `RedactingLoggerFactory` wraps the framework `LoggerFactory`; every `CreateLogger(name)` returns a `RedactingLogger` that intercepts `Log<TState>` and:
  - Rewrites the structured KV list, replacing values for known-sensitive keys (`password`, `accessToken`, `refreshToken`, `passwordResetToken`, `Authorization`, `Token`) with `***`.
  - Replaces the original raw values inside the formatted message string so the rendered template doesn't leak secrets either.
  - Wraps everything in a `RedactedState` that implements `IReadOnlyList<KeyValuePair<string, object?>>` so downstream providers (JsonConsole, the test capture) see the redacted state.
- **Replacement registration** — `services.Replace(ServiceDescriptor.Singleton<ILoggerFactory>(...))` builds the inner `LoggerFactory(providers, filterOptions)` and wraps it with `RedactingLoggerFactory`. This installs the decorator after the framework registers `LoggerFactory` so all providers receive redacted state.
- **Request-logging middleware** — `RequestLoggingMiddleware` runs after `SecurityHeaders` and before `ExceptionHandling`. For each request it:
  - Captures `traceId` from `Activity.Current?.TraceId` (falls back to `HttpContext.TraceIdentifier` when no activity is active).
  - Resolves `ICurrentUser.UserId` after the response is generated (so authenticated requests carry the user id).
  - Opens a `BeginScope` with `TraceId` + `UserId` so any user code logging during the request inherits these.
  - Emits `request.handled {Method} {Path} -> {Status} in {DurationMs}ms TraceId={TraceId} UserId={UserId}` at Information level.
