### Implementation

- **`CurrentUser`** model — `{ id, email, firstName, lastName, role }` mirroring the BT-014 `CurrentUserDto`.
- **`IMeService`** — two methods: `getMe(): Observable<CurrentUser>` (GET) and `deleteMe(): Observable<void>` (DELETE).
- **`MeService`** — concrete impl POSTs / DELETEs against `/api/me`. The `authInterceptor` automatically attaches the bearer token; the `refreshInterceptor` handles 401 retries. The composed effect: `dashboard.ngOnInit()` → `meApi.getMe()` → bearer token attached → BT-014 handler returns the user → dashboard `currentUser` signal populated → greeting computed reads from the live response.
- **`DashboardPage`** rewires the greeting source. `currentUser` (signal, defaults `null`) is set in `ngOnInit` from the `getMe()` response. The `email` and `role` computeds read `currentUser() ?? auth.snapshot()` so during the brief moment between sign-in navigation and the `getMe()` round-trip completing, the greeting still shows the cached sign-in values — no flicker, no blank state.

### Non-blocking observations

- `IMeService.deleteMe()` is shipped here but not yet called from any UI surface. FT-032 (the dashboard / profile delete-account flow) lands the consumer; until then, the BT-006 backend endpoint is reachable from the API client but unused from the app. Deferred per the FT-032 "ships with FT-031" pairing.
- The dashboard's error path swallows `getMe()` errors silently (the computed defaults fall back to `auth.snapshot()`). This keeps the dashboard from blanking on a transient network blip — auth-relevant 401s are caught one layer up by the `refreshInterceptor`. A future FI1.6 (error states) slice can surface a small inline indicator if `getMe` fails repeatedly.
- The BT-014 `CurrentUserDto` and the frontend `CurrentUser` model both stop at identity fields. BT-015 will expand the backend DTO with profile columns; the frontend model will need to grow alongside it. Documented here so the BT-015 implementer doesn't forget the frontend half.
- `MeService` doesn't use the `API_BASE_URL` token directly — it imports it from `auth.service.ts` (which exports `API_BASE_URL`). Keeps the token a single source of truth in the `api` lib. If a future slice wants per-resource base URLs, the token can move into its own file without breaking consumers.
