# L2 - Detailed Requirements

## L2-001: Local user registration
**Traces to:** L1-001

The system shall provide a registration form accepting first name, last name, email, and password, create a new user record on submission, hash the password before storage, and start an authenticated session on success.

**Acceptance Criteria:**
1. Given the user is on `/sign-up`, when they submit valid first name, last name, a unique email, and a password meeting policy, then a new `User` row is created with a hashed password and the user is redirected to the dashboard with a valid JWT.
2. Given a registration request with an email that already exists, when submitted, then the API responds 409 Conflict and no second account is created.
3. Given a registration request with a password shorter than 12 characters or missing one of {uppercase, lowercase, digit, symbol}, when submitted, then the API responds 400 with `ValidationProblemDetails` naming the failing fields.
4. Given the registration response succeeds, when the database is inspected, then the password column contains a hash from Argon2id, PBKDF2, or bcrypt — never plaintext.

---

## L2-002: Sign-in with username and password
**Traces to:** L1-001, L1-011

The system shall authenticate a user by email + password, issue a JWT access token and a refresh token on success, and reject invalid credentials with HTTP 401.

**Acceptance Criteria:**
1. Given a registered user and correct credentials, when they submit the sign-in form, then the API issues a signed JWT and the frontend stores it in memory for subsequent requests.
2. Given correct credentials and "Remember me" checked, when sign-in succeeds, then a refresh token is issued so the session survives a browser restart.
3. Given incorrect credentials, when submitted, then the API responds 401 with no information distinguishing "wrong email" from "wrong password".
4. Given a successful sign-in, when the JWT is decoded, then it contains issuer, audience, expiration ≤ 60 minutes, the user id, and the user's roles.

---

## L2-003: Sign-out
**Traces to:** L1-001, L1-011

The system shall let a signed-in user sign out from the profile screen, invalidate the active refresh token server-side, and return them to the sign-in screen.

**Acceptance Criteria:**
1. Given a signed-in user, when they tap "Sign out" on the profile screen, then the refresh token is revoked server-side and the access token is cleared client-side.
2. Given a signed-out session, when the user attempts to access an authenticated route, then they are redirected to `/sign-in`.

---

## L2-004: Password reset via email link
**Traces to:** L1-001

The system shall let a user request a password reset by email; the system shall log a reset link (no real email integration in MVP — a logging no-op service stands in for transactional email and is documented as such), accept the link to set a new password, and invalidate prior sessions on success.

**Acceptance Criteria:**
1. Given any email address submitted at `/password-reset`, when submitted, then the API responds 202 Accepted regardless of whether the email exists (no account enumeration).
2. Given a registered email, when the reset is requested, then a single-use, time-limited (≤ 30 minutes) token is generated and logged via the email no-op service.
3. Given a valid reset token, when the user submits a new password meeting policy, then the password is hashed and stored, all existing refresh tokens for that user are revoked, and the user is redirected to sign in.
4. Given an expired or already-used token, when submitted, then the API responds 400 and no password change occurs.

---

## L2-005: Profile management
**Traces to:** L1-001

The system shall let a user view and edit first name, last name, email, units (Imperial / Metric), time zone, daily active-calorie target, daily workout-minutes target, and current weight from the profile screen.

**Acceptance Criteria:**
1. Given a signed-in user on `/profile`, when they change any editable field and submit, then the new value persists across sign-out / sign-in.
2. Given the user changes email, when submitted, then the API enforces uniqueness and the next sign-in requires the new email.
3. Given viewport width < 576px (XS), when the profile screen renders, then field rows collapse to a single column so values like `America/Toronto` are not truncated.

---

## L2-006: Account deletion
**Traces to:** L1-001, L1-017

The system shall let a user permanently delete their account; on deletion the system shall remove or anonymize all PII for that user.

**Acceptance Criteria:**
1. Given a signed-in user, when they confirm "Delete account" twice, then the API soft-deletes the row, anonymizes name + email, revokes all tokens, and signs them out.
2. Given a deleted account, when anyone attempts to sign in with that email, then the API responds 401 as if the account never existed.

---

## L2-007: Log workout session manually
**Traces to:** L1-002

The system shall let a user create a workout session capturing equipment type, date, start time, duration (minutes), distance (where applicable), average heart rate, active calories, and notes, then persist it to the database.

**Acceptance Criteria:**
1. Given a signed-in user on `/workouts/new`, when they submit a session with equipment = `Treadmill`, duration = 22, distance = 2.1, avg HR = 128, active calories = 218, then a row is created and the dashboard ring updates within 1 second.
2. Given duration ≤ 0 or > 480, when submitted, then the API responds 400 via FluentValidation naming the duration field.
3. Given equipment = `Bench Press`, when submitted, then the distance field is hidden in the form and not required.

---

## L2-008: List and filter workout sessions
**Traces to:** L1-002

The system shall list past workout sessions in reverse chronological order with filters by equipment type, date range (today / this week / this month / all), and search by notes substring.

**Acceptance Criteria:**
1. Given the user has logged 10 sessions across all four equipment types, when they tap "Treadmill" filter, then only treadmill sessions remain visible.
2. Given the user is on `/workouts` at viewport ≥ 1200px (XL), when the list renders, then the filter chips remain visible without horizontal scroll.
3. Given viewport < 576px (XS), when the filter chip strip overflows, then it scrolls horizontally inside its container without forcing the page to scroll horizontally.

---

## L2-009: Edit, duplicate, or delete a workout session
**Traces to:** L1-002

The system shall let a user open a session on `/workouts/:id`, edit any field, duplicate the session as a new row, or delete the session. Deletion shall refund any points awarded for that session.

**Acceptance Criteria:**
1. Given a session that earned +85 points, when the user deletes it, then their points balance decreases by exactly 85 within 1 second.
2. Given the user duplicates a session, when the new row is created, then date defaults to today, all other fields copy from the source, and the new row appears at the top of `/workouts`.

---

## L2-010: Supported equipment types
**Traces to:** L1-002

The system shall recognize exactly four equipment types — treadmill, indoor bike, bench press, elliptical — and expose them via a `GET /equipment` endpoint backed by a static seeded table.

**Acceptance Criteria:**
1. Given a fresh database, when the seed runs, then the `Equipment` table contains exactly four rows: `treadmill`, `indoor-bike`, `bench-press`, `elliptical`.
2. Given a session-create request with `equipment = "rowing-machine"`, when submitted, then the API responds 400 with a validation error.

---

## L2-011: Daily active-calories ring with 1,500 kcal default target
**Traces to:** L1-003

The system shall display the user's active calories burned today against their target on the dashboard; the default target shall be 1,500 kcal and shall be editable on the profile screen.

**Acceptance Criteria:**
1. Given the user has burned 980 active kcal today, when the dashboard renders, then the ring fills to 980 / 1500 (≈65%) and the numeric label reads `980 / 1500 kcal`.
2. Given the user changes the daily target to 1,800 on the profile screen, when they return to the dashboard, then the ring denominator updates to 1,800 without a manual refresh.
3. Given viewport ≥ 992px (LG), when the dashboard renders, then the ring sits in a two-column layout next to the streak summary; at < 768px (XS / SM) the layout collapses to a single column.

---

## L2-012: Daily total workout minutes
**Traces to:** L1-003

The system shall display total minutes of workout time logged today on the dashboard, summed across every equipment type.

**Acceptance Criteria:**
1. Given the user has logged sessions of 22 + 18 + 30 minutes today, when the dashboard renders, then the "minutes today" tile reads `70 min`.
2. Given the user has logged zero sessions today, when the dashboard renders, then the tile reads `0 min` and the empty-state CTA invites them to start a workout.

## L2-013: Authenticated dashboard summary
**Traces to:** L1-003, L1-006

The system shall present a dashboard at `/` after sign-in summarizing today's calorie progress, today's minutes, current streak, current points balance, current tier, and the next reward within reach.

**Acceptance Criteria:**
1. Given a signed-in user, when they navigate to `/`, then the dashboard loads in ≤ 1.5 s on a 4G profile and shows all six summary tiles above the fold at 1440px width.
2. Given the user is unauthenticated, when they navigate to `/`, then they are redirected to `/sign-in`.

## L2-014: Monthly weight-loss goal
**Traces to:** L1-004

The system shall let the user set a monthly weight-loss goal (default −20 lb / month) on the profile screen and shall display progress on the dashboard.

**Acceptance Criteria:**
1. Given a goal of −20 lb / month and the user has logged a 5.2 lb loss so far this month, when the dashboard renders, then the goal tile reads `−20 lb / month · On track · −5.2 lb so far`.
2. Given a goal of 0 or > 30 lb / month, when submitted, then the API responds 400 via FluentValidation.

## L2-015: Current-weight tracking
**Traces to:** L1-004

The system shall let the user record current body weight on the profile screen; the value shall be timestamped and stored historically so the dashboard can compute month-to-date progress.

**Acceptance Criteria:**
1. Given the user enters a new current weight, when saved, then a row is appended to `WeightEntry` with the current UTC timestamp, and prior entries are preserved.
2. Given two entries on the same day, when stored, then both are persisted; the dashboard uses the most recent.

## L2-016: Morning workout window
**Traces to:** L1-005

The system shall let the user configure a morning window (default 5:00–7:30) used to award the morning-bonus points multiplier and trigger the morning reminder.

**Acceptance Criteria:**
1. Given the morning window is 5:00–7:30 and the user logs a session at 5:12, when scoring runs, then the session earns the +25 morning bonus.
2. Given the user changes the window end to 8:00, when they save, then a session at 7:50 earns the morning bonus.

---

## L2-017: Kitchen-closed (night-eating) window
**Traces to:** L1-005

The system shall let the user configure a kitchen-closed window (default 20:00–06:00) used to trigger the kitchen-closes nudge and to score the daily "no late eating" badge.

**Acceptance Criteria:**
1. Given a kitchen-closed window of 20:00–06:00, when the local clock crosses 19:45, then the kitchen-closes reminder fires (subject to L2-026).
2. Given the user toggles the kitchen-closes nudge off on profile, when 19:45 arrives the next day, then no notification is delivered.

---

## L2-018: Base points for logging a session
**Traces to:** L1-006

The system shall award 2 base points per minute of logged workout, computed at session-create time and persisted in a `PointsLedger`.

**Acceptance Criteria:**
1. Given a 22-minute session, when created, then a `PointsLedger` row is appended with `+44 (Base — 22 min logged)`.
2. Given a session deletion, when the row is removed, then a compensating `−44 (Refund)` ledger row is appended.

---

## L2-019: Morning bonus points
**Traces to:** L1-006

The system shall award +25 bonus points to any session that starts inside the morning window configured per L2-016.

**Acceptance Criteria:**
1. Given a session starting at 05:12 inside a 05:00–07:30 window, when scoring runs, then a `+25 (Morning bonus before 7 AM)` row is appended.
2. Given a session starting at 09:00, when scoring runs, then no morning-bonus row is appended.

---

## L2-020: Streak multiplier
**Traces to:** L1-006

The system shall track consecutive-day workout streaks and apply a multiplier ≥ 1.00 (rounded to two decimals) to each session's awarded points.

**Acceptance Criteria:**
1. Given the user has worked out 7 consecutive days, when scoring runs for today's session, then the multiplier is 1.07 and an additional `+6 (Streak multiplier ×1.07)` row is appended for a 22-minute session.
2. Given the user misses one day, when they next log a session, then the streak resets to 1 and the multiplier returns to 1.00.

---

## L2-021: Rewards catalog and redemption
**Traces to:** L1-006

The system shall expose a static rewards catalog (e.g., post-workout smoothie, rest-day pass, new running socks tier) on `/rewards`; the user shall redeem rewards by spending points.

**Acceptance Criteria:**
1. Given the user has 1,250 points and the smoothie reward costs 500, when they tap "Redeem", then the balance becomes 750 and a `RewardRedemption` row is created.
2. Given the user has 200 points and the reward costs 500, when they tap "Redeem", then the API responds 400 with `INSUFFICIENT_POINTS` and the balance is unchanged.

---

## L2-022: Tier progression
**Traces to:** L1-006

The system shall compute a user tier (Bronze / Silver / Forged Iron / Gold / Platinum) from cumulative lifetime points and display it on the profile and dashboard.

**Acceptance Criteria:**
1. Given the tier thresholds are deterministic, when a user crosses 5,000 lifetime points, then their tier becomes `Forged Iron` and the profile shows `Tier 3 — Forged Iron`.
2. Given a redemption refund returns a user below a threshold, when the next dashboard load occurs, then the displayed tier reflects the lower value.

---

## L2-023: Apple Watch / HealthKit sync
**Traces to:** L1-007

The system shall pull active calories, heart rate, and workout sessions from HealthKit at a configurable cadence (default every 4 minutes when authorized), reconcile imported sessions against manual entries, and mark imports with `source = "AppleWatch"`.

**Acceptance Criteria:**
1. Given HealthKit authorization and a workout recorded on the watch, when the next sync runs, then a `Session` row is created with `source = "AppleWatch"` and the dashboard updates without user action.
2. Given a manual session and an imported session that overlap by ≥ 50% on time and equipment, when reconciled, then the imported session is merged into the manual one and not duplicated.

---

## L2-024: Apple Watch sync error and retry
**Traces to:** L1-007, L1-009

The system shall display a sync-error screen at `/error` describing the failure, the diagnostic checks (servers, internet, HealthKit auth, watch reachability), and offer "Retry sync" + "Go to dashboard" actions.

**Acceptance Criteria:**
1. Given HealthKit returns `errAuthorizationDenied`, when the next sync runs, then the user sees the inline "Apple Watch sync paused" banner with a `Retry now` link and the body shows `# ERR_HEALTHKIT_OFFLINE`.
2. Given the user taps "Retry sync" and HealthKit succeeds, then the banner disappears and the dashboard ring updates with the freshly imported calories.

---

## L2-025: Morning workout reminder
**Traces to:** L1-008

The system shall send a morning workout reminder at the start of the configured morning window if the user has not yet logged a session today.

**Acceptance Criteria:**
1. Given the morning window starts at 05:00 and no session has been logged today, when the local clock reaches 04:55, then the reminder fires.
2. Given the user has already logged a session before the window start, when the window opens, then the reminder is suppressed.

---

## L2-026: Kitchen-closes nudge
**Traces to:** L1-008

The system shall send a kitchen-closes nudge 15 minutes before the configured kitchen-closed window start, contingent on user opt-in.

**Acceptance Criteria:**
1. Given a 20:00 kitchen-closed start and the nudge is enabled, when the clock reaches 19:45, then a notification fires.
2. Given the nudge is disabled on profile, when 19:45 arrives, then no notification is delivered.

---

## L2-027: Friend leaderboard opt-in
**Traces to:** L1-008

The system shall let the user opt in to a friends leaderboard from profile; while opted out, the user's stats shall not appear on any other user's leaderboard.

**Acceptance Criteria:**
1. Given user A has opted out, when user B loads the leaderboard, then user A's row is absent.
2. Given user A toggles opt-in to true, when user B refreshes, then user A appears with current points.

## L2-028: Empty-state surfaces
**Traces to:** L1-009

The system shall present an empty state on every list surface (`/workouts`, `/rewards`, leaderboard) explaining the missing data and offering the primary CTA.

**Acceptance Criteria:**
1. Given the user has logged zero sessions, when they open `/workouts`, then they see an illustration, the copy "No workouts yet", and a primary "Log first workout" button.
2. Given any empty state at viewport < 576px, when rendered, then the illustration scales to fit and the CTA reaches the bottom of the visible area as a full-width button.

## L2-029: Error-state surface for unrecoverable failures
**Traces to:** L1-009

The system shall route any unrecoverable client-side failure to `/error` with diagnostics and a "Retry" / "Go to dashboard" recovery affordance.

**Acceptance Criteria:**
1. Given an API call returns 5xx after retry, when the failure surfaces, then the user lands on `/error` with the trace id visible (e.g., `Trace ID: 7c3a-fae9-1180`) for support.
2. Given the user taps "Go to dashboard", when navigation completes, then they return to `/` with no residual error banner.

---

## L2-030: Responsive breakpoints across XS through XL
**Traces to:** L1-010

The system shall reflow every authenticated and unauthenticated surface across XS (<576px), SM (≥576px), MD (≥768px), LG (≥992px), and XL (≥1200px) without horizontal scrolling, clipped content, or overlapping elements.

**Acceptance Criteria:**
1. Given any surface, when rendered at 360px width, then no element forces horizontal scroll on the document.
2. Given the dashboard, when rendered at ≥ 1200px width, then the navigation switches from the bottom bar to a left navigation rail.
3. Given the workout-detail screen, when rendered at any of 360 / 768 / 1440, then the Notes textarea displays the full multi-line example without clipping.

---

## L2-031: Password storage
**Traces to:** L1-011, L1-017

The system shall store user passwords only as salted hashes from a modern password-hashing function (Argon2id, PBKDF2 with ≥ 600,000 iterations, or bcrypt with cost ≥ 12). Plaintext passwords shall never be persisted, returned, or logged.

**Acceptance Criteria:**
1. Given a registration request, when the row is persisted, then the `PasswordHash` column matches a recognizable Argon2id / PBKDF2 / bcrypt format and contains a per-user salt.
2. Given any log line emitted during sign-in or registration, when inspected, then no plaintext password value appears.

---

## L2-032: JWT issuance and validation
**Traces to:** L1-011

The system shall issue HS256- or RS256-signed JWTs with explicit `iss`, `aud`, `exp`, `nbf`, `iat`, and `sub` claims, validate every claim on every request, and reject invalid tokens with HTTP 401.

**Acceptance Criteria:**
1. Given a JWT with a tampered signature, when sent on any authenticated request, then the API responds 401 and emits an audit log row.
2. Given a JWT with `exp` in the past, when sent, then the API responds 401 with `WWW-Authenticate: Bearer error="invalid_token"`.

---

## L2-033: Refresh tokens
**Traces to:** L1-011

The system shall issue opaque, single-use, server-stored refresh tokens that rotate on use; reuse of a consumed refresh token shall revoke the entire token family for that user.

**Acceptance Criteria:**
1. Given a valid refresh token, when exchanged, then a fresh access token + refresh token pair is issued and the old refresh token is marked consumed.
2. Given a previously-consumed refresh token, when replayed, then the API responds 401 and the user's entire refresh-token family is revoked.

---

## L2-034: Failed sign-in throttling and lockout
**Traces to:** L1-011

The system shall throttle repeated failed sign-ins by IP and by email; after 5 consecutive failures within 15 minutes for a given email, the account shall be temporarily locked for 15 minutes.

**Acceptance Criteria:**
1. Given 5 consecutive failed sign-ins for `user@example.com`, when the 6th is attempted within 15 minutes, then the API responds 429 even if the password is correct.
2. Given a successful sign-in resets the counter, when the user signs in correctly after 4 failures, then the counter is cleared.

---

## L2-035: Security audit log
**Traces to:** L1-011, L1-015

The system shall write a structured audit-log entry for every sign-in attempt (success or failure), every password reset, every account deletion, and every refresh-token revocation, with user id, IP, user agent, and timestamp.

**Acceptance Criteria:**
1. Given a failed sign-in, when the request completes, then exactly one `AuditLog` row is created with `event = "sign-in.failure"` and IP populated.
2. Given a successful sign-in, when the request completes, then one `AuditLog` row is created with `event = "sign-in.success"`.

---

## L2-036: PKCE and external IdPs are out of scope
**Traces to:** L1-011

The system shall not surface PKCE flows or external identity-provider buttons (Apple, Google, GitHub, etc.) anywhere in the UI or API.

**Acceptance Criteria:**
1. Given any unauthenticated screen, when rendered, then no Apple / Google / GitHub sign-in buttons are present.
2. Given a grep of the codebase for "PKCE" or "OAuth provider", when run, then no implementation results are returned.

---

## L2-037: Role definitions
**Traces to:** L1-012

The system shall define exactly two roles for the MVP: `User` (default for all registered accounts) and `Admin` (granted manually via database seed or admin tool).

**Acceptance Criteria:**
1. Given a freshly registered user, when their record is inspected, then they hold the `User` role and only the `User` role.
2. Given an admin-only endpoint, when called by a `User`-role JWT, then the API responds 403.

---

## L2-038: Authorization on every endpoint
**Traces to:** L1-012

The system shall require an authenticated JWT on every endpoint except sign-up, sign-in, password-reset request/confirm, and the health endpoint; admin endpoints shall require the `Admin` role.

**Acceptance Criteria:**
1. Given an unauthenticated request to `GET /workouts`, when sent, then the API responds 401.
2. Given a `User`-role JWT request to `GET /admin/users`, when sent, then the API responds 403.

---

## L2-039: FluentValidation per command and DTO
**Traces to:** L1-013

Every command, query, and request DTO shall have a colocated `AbstractValidator<T>` registered by assembly scan; a MediatR pipeline behavior shall run validators before each handler. Request DTOs shall not use `System.ComponentModel.DataAnnotations` attributes.

**Acceptance Criteria:**
1. Given a `CreateSessionCommand`, when the assembly is scanned, then a `CreateSessionCommandValidator` is registered and invoked before the handler.
2. Given a request DTO, when grepped for `[Required]`, `[StringLength]`, etc. from `System.ComponentModel.DataAnnotations`, then no matches are found.

---

## L2-040: ValidationProblemDetails on validation failure
**Traces to:** L1-013

The system shall map validator failures to HTTP 400 with a `ValidationProblemDetails` body listing each failing field and message.

**Acceptance Criteria:**
1. Given a `CreateSessionCommand` with `duration = 0`, when posted, then the API responds 400 with `errors.duration` populated and content-type `application/problem+json`.
2. Given a request that fails three validators across two fields, when posted, then the response lists all three failure messages keyed by field.

---

## L2-041: API response-time targets
**Traces to:** L1-014

The system shall meet the following P95 response-time targets under nominal load (≤ 50 RPS): authentication endpoints ≤ 300 ms, read endpoints ≤ 200 ms, write endpoints ≤ 400 ms.

**Acceptance Criteria:**
1. Given a load test at 50 RPS for 5 minutes against `GET /sessions`, when measured, then P95 is ≤ 200 ms and error rate is 0.
2. Given a load test at 50 RPS against `POST /sessions`, when measured, then P95 is ≤ 400 ms.

---

## L2-042: Dashboard initial-render performance
**Traces to:** L1-014

The system shall render the dashboard's first contentful paint in ≤ 1.5 s and time-to-interactive in ≤ 2.5 s on a Lighthouse "Slow 4G" mobile profile.

**Acceptance Criteria:**
1. Given a fresh authenticated session and a cold cache, when Lighthouse mobile audit runs against `/`, then FCP ≤ 1.5 s and TTI ≤ 2.5 s.
2. Given the same audit on desktop, when run, then FCP ≤ 1.0 s.

---

## L2-043: Structured logging
**Traces to:** L1-015

The system shall emit logs in structured JSON via `Microsoft.Extensions.Logging` with at minimum `timestamp`, `level`, `message`, `traceId`, and `userId` (when known) fields.

**Acceptance Criteria:**
1. Given any handled request, when a log line is emitted, then it parses as valid JSON with the five required fields populated.
2. Given an unhandled exception, when caught by middleware, then a single error-level log line is emitted with the stack trace and trace id.

## L2-044: Health endpoint
**Traces to:** L1-015

The system shall expose a `GET /health` endpoint returning HTTP 200 with `{ "status": "Healthy" }` when the database is reachable and HTTP 503 otherwise; the endpoint shall not require authentication.

**Acceptance Criteria:**
1. Given a running app and reachable database, when `GET /health` is called, then the response is 200 with `status = "Healthy"`.
2. Given the database connection is severed, when `GET /health` is called, then the response is 503 with a populated `status` other than `"Healthy"`.

**Acceptance Criteria:**

**Traces to:** L1-016

Every text element shall present a contrast ratio of ≥ 4.5:1 against its background (≥ 3:1 for large text ≥ 18.66px / 24px bold).

**Acceptance Criteria:**
1. Given the dashboard ring numeric label, when measured against the surface, then contrast is ≥ 4.5:1.
2. Given the green hero on the sign-in screen, when measured against the white headline, then contrast is ≥ 4.5:1.

## L2-049: PII protection at rest and in transit
**Traces to:** L1-017

The system shall serve every endpoint over TLS 1.2+ in non-development environments and shall not store sensitive PII (passwords, tokens) in plaintext at rest.

**Acceptance Criteria:**
1. Given the production deployment, when an HTTP (port 80) request is sent, then it is redirected to HTTPS or rejected.
2. Given a database dump, when grepped for plaintext password values from a known test user, then no matches are found.

---

## L2-050: Account deletion clears user data
**Traces to:** L1-017, L1-001

When a user deletes their account, the system shall delete or anonymize PII fields (first name, last name, email) within 24 hours and shall preserve only aggregate, non-identifying ledger data needed for system integrity.

**Acceptance Criteria:**
1. Given a deleted account, when the user record is inspected after the next nightly cleanup job, then `Email`, `FirstName`, and `LastName` are null or hashed sentinel values.
2. Given audit-log entries that reference the deleted user, when inspected, then the `userId` field remains for trace integrity but no PII is present.

---

## L2-051: No plaintext logging of secrets
**Traces to:** L1-017

The system shall redact passwords, tokens, refresh tokens, and reset tokens from all log output and error responses.

**Acceptance Criteria:**
1. Given a sign-in request body in any log line, when inspected, then the password field reads `[REDACTED]` (or is absent).
2. Given an exception thrown from auth middleware, when serialized into the response, then no Authorization header value is included.
