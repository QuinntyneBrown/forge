App Delivery Workflow

Audience: Claude agent only. This document is the single source of truth for what to do next. Do not rewrite it for human readability.

Mission: Drive ./docs/idea.md to a deployed app using this fixed, evaluable, parallelizable workflow.

Canonical location: ./docs/dotnet-angular-authenticated-full-stack-workflow.html. This is the only copy. Every status / owner update the agent makes is to this path. Do not create or maintain a copy outside ./docs/.

Loop command: /loop 1m ./docs/dotnet-angular-authenticated-full-stack-workflow.html ./docs/idea.md

Agent Protocol (run this every loop iteration)

Multi-agent coordination: Two or more agents may be running this workflow on different machines. ./docs/dotnet-angular-authenticated-full-stack-workflow.html in origin/main is the shared lock table. Every status or owner change MUST be committed and pushed immediately so the other agent sees it on its next iteration. Identify yourself with a stable, unique data-owner value — for example claude@<hostname>. Never use the bare value claude when more than one agent is active.

  1. Sync from remote first. Run git fetch origin then git pull --rebase origin main in the repo root. Do not read ./docs/dotnet-angular-authenticated-full-stack-workflow.html until the pull succeeds. If the working tree is dirty (uncommitted edits from a prior aborted iteration), git stash them, pull, then git stash pop and resolve before continuing.
  2. Read this file and parse every <article data-task-id="..."> element.
  3. Resolve statuses: for any task whose data-status="blocked", recheck dependencies. If every ID in data-depends-on has data-status="done" (or the list is empty), update that task's data-status to ready. If you flipped any statuses, save, commit (chore(workflow): unblock <ids>), and push before continuing.
  4. Pick the next task using this priority order:
    1. Any task already data-status="in-progress" with data-owner equal to your identifier — resume it.
    2. Otherwise, the first task in document order with data-status="ready" and data-owner="".
    3. If no ready task exists and any task is still blocked or in-progress by another owner, exit this iteration (wait for the next loop tick).
    4. If every task has data-status="done", the workflow is complete — write WORKFLOW COMPLETE and stop the loop.
  5. Claim it atomically (this is the critical section for two-agent safety):
    1. Set data-status="in-progress" and data-owner="<your-id>" on the chosen article. Save.
    2. git add docs/dotnet-angular-authenticated-full-stack-workflow.html
    3. git commit -m "claim(workflow): <task-id> by <your-id>"
    4. git push origin main
    5. If the push is rejected (non-fast-forward), the other agent claimed something concurrently. git reset --hard origin/main after fetching, then restart this iteration from step 1. Do not retry the same task without re-reading.
    The push-success is the lock acquisition. Until it succeeds, you do not own the task.
  6. Execute the task per its Goal, Inputs, and Deliverable. Use the skill named in data-skill if one exists; otherwise proceed without a skill. Commit and push deliverable artifacts (code, docs, tests) as you produce them, on their own commits separate from workflow file state changes.
  7. Evaluation tasks (data-evaluation-passes > 0): run the evaluation that many times, appending findings to ./docs/evaluations/<task-id>.md before marking done. Implementation tasks use ATDD: write the acceptance test first, then make it pass.
  8. On completion:
    1. Set data-status="done" on the article and clear data-owner.
    2. Re-run step 3 so newly unblocked tasks flip to ready in the same edit.
    3. Save, then git add docs/dotnet-angular-authenticated-full-stack-workflow.html, git commit -m "done(workflow): <task-id>", git push origin main.
    4. If push is rejected: git pull --rebase origin main, reapply your status change on top (your done still wins because you held the lock), then push again.
  9. If blocked mid-task, leave data-status="in-progress", append a note to a sibling <p data-note="blocker"> inside the article describing what is needed. Save, commit (note(workflow): <task-id> blocker), push, and exit the iteration. The other agent must not steal an in-progress task; only the owner clears it.
  10. End of iteration: ensure your working tree is clean and your local main matches origin/main. If not, you have unpushed work — push it before the loop fires again.

Concurrency rules:

  • Tasks sharing a data-parallel-group may proceed independently once dependencies are met. With two agents, the second one picks the next ready task in document order — usually the parallel sibling.
  • Only the owner of an in-progress task may modify its data-status or data-owner. Never reassign someone else's task.
  • If you find a task in-progress by another owner that has had no commits in 30 minutes (check git log -1 --format=%cI -- docs/dotnet-angular-authenticated-full-stack-workflow.html), append a <p data-note="stale-owner"> note but do not claim it. A human will resolve.
  • Push-rejection on a workflow-file commit is the expected concurrency signal — handle it, do not panic. Push-rejection on a deliverable commit is a normal merge — pull-rebase and re-push.

Never delete tasks, reorder articles, modify data-task-id, force-push main, or change human-readable wording outside the data-* attributes and the data-note elements you append.

Attribute Legend

  • data-task-id: stable identifier referenced by data-depends-on.
  • data-status: ready | blocked | in-progress | done.
  • data-owner: unique agent identifier currently holding the task (e.g. claude@laptop-01); empty when unclaimed.
  • data-parallel-group: tasks sharing a group can run together when their dependencies are met.
  • data-depends-on: comma-separated task IDs that must be done before this task starts.
  • data-skill: the Claude skill to invoke for this task (or empty for none).
  • data-evaluation-passes: required number of evaluation iterations (0 = not an evaluation task).

Implementation Guidance (mandatory for every backend / frontend task)

This section is the only technology specification the agent will receive. Every implementation task (MB1, MF1, BP1, BT1, BI1, FP1, FT1, FI1) and every implementation evaluation (MB2, MF2, BP2, BT2, FP2, FT2, plus the per-slice eval inside BI1/FI1) MUST be executed against and verified against these rules. References below to "the guidance" mean this section.

General

  • Mobile-first web app that also looks great and works well on large screens.
  • The entire solution shall follow S.O.L.I.D. principles.
  • Implementation is radically simple: the smallest design that fully implements the requirements. No speculative abstractions, no half-built features, no dead code paths.
  • No stubs, no TODO placeholders, no throw new NotImplementedException, no console.log("TODO"), no commented-out logic left "for later". Within reason: optional integrations explicitly deferred in ./docs/idea.md or the requirements (for example, transactional email sending) may be replaced by a clearly-named no-op service that logs the intended action; this is the only acceptable form of deferred work and must be called out in the deliverable notes.
  • Every requirement listed in the approved requirements document must be implemented in its entirety before its task is marked done.

Backend (.NET)

  • Clean Architecture (Api / Application / Domain / Infrastructure separation).
  • Command/Query Separation (CQS) using MediatR (free version). Commands and queries live in the Application layer.
  • ASP.NET Core Controllers for HTTP endpoints (no minimal APIs unless the requirement explicitly demands it).
  • Entity Framework Core DbContext for data access.
  • Command and query handlers depend on an IAppDbContext interface. The concrete AppDbContext inherits DbContext and implements IAppDbContext. Replace App with the solution-specific name.
  • Do not add repository or unit-of-work abstractions over EF Core. Handlers use IAppDbContext directly.
  • Microsoft SQL Server as the database.
  • Use Microsoft.Extensions.* for Logging, Configuration, and Dependency Injection.
  • Old-style solution file format (.sln):
    • backend/src — Api project, Application, Domain, Infrastructure, etc.
    • backend/tests — backend tests.
  • One type per C# file. Every class, interface, enum, record, struct, and delegate gets its own file. NO files containing a class AND an interface, multiple enums, multiple classes, or any combination. The file name MUST match the type name (e.g., FooService.cs contains FooService and nothing else).

Validation

  • Use FluentValidation. Do not use System.ComponentModel.DataAnnotations attributes on commands, queries, or request DTOs.
  • One AbstractValidator<TCommand> per command, colocated with the command in the Application layer (<CommandName>Validator.cs). Same pattern for nested DTOs.
  • A MediatR pipeline behavior runs validators before the handler; failures throw ValidationException and the API maps them to HTTP 400 ValidationProblemDetails.
  • Register validators by assembly scanning so new ones are picked up automatically.
  • HTTP request DTOs are pure transport shapes — no validation attributes; map to commands and let the Application layer validate.

Authentication & User Management

  • OAuth 2.0 sign-in with local users: backend stores users in the application database, validates username + password credentials, and issues a signed JWT access token (and a refresh token where appropriate). The frontend consumes the JWT and attaches it to API requests as a bearer token. PKCE and external identity providers are explicitly out of scope for this workflow.
  • Passwords stored only as salted hashes from a modern password-hashing function (Argon2id, PBKDF2 with adequate iterations, or bcrypt with adequate cost). Never store or log plaintext passwords, code verifiers, or tokens.
  • JWTs validated on every request: issuer, audience, signature, expiration. Invalid/expired tokens → HTTP 401.
  • Repeated failed sign-ins are rate-limited, locked out, or otherwise throttled, with a security audit log entry on each event.
  • Full user management — registration, sign-in, sign-out, password reset, profile management, account deletion.
  • RBAC implementation from database to frontend.

Frontend (Angular)

  • Angular Material components for every component — buttons, headers, inputs, etc. Visual language is Material 3 (https://m3.material.io/).
  • Design tokens for consistent colors and spacing across the app.
  • BEM naming convention for all HTML/CSS classes.
  • One type per file. All components MUST be split into .html, .scss, .ts file triples. NO single-file components.
  • Angular workspace split into libraries and apps.
  • Libraries use interface-driven service consumption (see https://github.com/QuinntyneBrown/interface-driven-service-consumption).

Library Structure

  • api library — models and services needed to communicate with the backend.
    • Exposes an interface for each service:
      • foo.service.ts
      • foo.service.contract.ts — includes the TypeScript interface and the injection token.
  • components library — reusable UI components (buttons, headers, etc.). Does not depend on api library.
  • domain library — UI components that depend on api library.
    • Exposes an interface for each service (if any):
      • foo.service.ts
      • foo.service.contract.ts — includes the TypeScript interface and the injection token.
  • main application — depends on api, components, and domain libraries.

Testing

  • E2E testing using Playwright Page Object Model for important functionality. ATDD: write the Playwright POM acceptance test first, then implement until it passes.

Implementation Evaluation Rubric (use for every implementation eval task)

Every implementation evaluation (MB2, MF2, BP2, BT2, FP2, FT2, and the per-slice eval inside BI1 and FI1) MUST verify each of the following and record findings under ./docs/evaluations/<task-id>.md. Any failure is blocking — fix it in the implementation, then re-run the evaluation pass.

  1. Guidance adherence. Open the changed code and confirm each relevant bullet from the sections above is satisfied. Backend evals check the Backend, Validation, Auth, and General sections; frontend evals check the Frontend, Library Structure, Auth (frontend side), and General sections; both check Testing.
  2. Requirements coverage in entirety. Every requirement in scope for this task is fully implemented end to end (UI → API → DB where applicable). No "partial" features.
  3. Radically simple. No speculative abstractions, no extra layers, no dead code, no commented-out experiments, no unused parameters, no premature interfaces, no defensive guards for impossible states.
  4. No temp code or stubs. Grep for TODO, FIXME, XXX, HACK, NotImplementedException, throw new Error("not implemented"), empty method bodies that should do work, returned hard-coded sentinel values, commented-out blocks. Each hit is a finding unless it is an explicitly-deferred optional integration documented in the deliverable notes (e.g., transactional email — replaced by a logging no-op service named accordingly).
  5. One type per file. No combined-type C# files; no single-file Angular components.
  6. SOLID + CQS shape. Backend: handlers depend on IAppDbContext, FluentValidation present for every command, no repository/unit-of-work classes. Frontend: services are interface-driven via *.service.contract.ts, libraries respect the dependency direction (components → nothing; domain → api; app → all).
  7. Frontend library placement (CRITICAL — every frontend evaluation must enforce this). Walk every Angular file and verify it lives in the correct library:
    • Every reusable presentation component (buttons, headers, inputs, cards, dialogs, layout primitives, anything that does not call the backend) lives in the components library. If a presentation component is found anywhere else (in domain, in app, in api), it is a finding. Move it to components.
    • Every model and every service used to communicate with the backend (HTTP services, DTO interfaces, response/request types, the matching *.service.contract.ts + injection token) lives in the api library. If a backend-talking service or its model is found in components, domain, or app, it is a finding. Move it to api.
    • Every component that needs the api layer (any component that injects a service exposed by api, directly or transitively) lives in the domain library. If such a component is found in components or app, it is a finding. Move it to domain.
    • Verification: grep imports. components must import nothing from api or domain. domain may import from api and components, never from app. app may import from all three. Any violation is blocking.
  8. ATDD evidence. Acceptance test exists and was written before the implementation (check git history if needed). It currently passes against the actual implementation.
  9. Mobile-first + responsive. Frontend changes look correct at 360 / 768 / 1440 widths.
  10. Build and run clean. dotnet build and ng build (or equivalent) produce zero errors and zero new warnings. App starts from the documented run command.

Fix-on-find rule: if any evaluation pass surfaces a finding, the same agent fixes it in the implementation, commits and pushes the fix, then runs another evaluation pass. The eval task is not done until a complete pass produces zero findings.

R1 - Create requirements (L1 + L2)

Goal: Translate ./docs/idea.md and the accepted mocks under ./docs/mocks/ into structured, traceable requirements that drive ATDD. Produce two files: ./docs/specs/L1.md (high-level capabilities) and ./docs/specs/L2.md (detailed, testable behaviors with Given/When/Then acceptance criteria). This task inlines the requirements-engineer policy below — that section is the only requirements spec available.

Inputs: ./docs/idea.md, accepted mocks under ./docs/mocks/, Implementation Guidance

Deliverable: two files under ./docs/specs/:

  • ./docs/specs/L1.md — high-level requirements
  • ./docs/specs/L2.md — detailed requirements with acceptance criteria, each tracing to an L1
Create the ./docs/specs/ directory if it does not exist. If L1.md or L2.md already exist, read them first and update rather than overwrite — preserve existing requirement IDs, only add / modify / remove entries.

Core philosophy. Requirements exist so every line of code traces back to a deliberate decision. Without them, teams build the wrong thing, skip edge cases, and ship insecure software. The flow is:

  1. L1 (High-Level) — what the system must do, stated broadly.
  2. L2 (Detailed) — specific, testable behaviors, each tracing to an L1.
  3. Acceptance Tests — written before implementation (during BI1 / FI1), each tracing to one or more L2s.
No code gets written until there is a failing acceptance test. No acceptance test is written without an L2 backing it.

Default assumptions (assume all of these unless ./docs/idea.md explicitly says otherwise):

  • Production-grade software: account for reliability, maintainability, observability, and operational concerns. Not a prototype, not a toy.
  • Security-first: every feature considers authentication, authorization, input validation, data protection, and the OWASP Top 10. Add security requirements even when the idea does not mention them.
  • High performance: address response times, throughput, resource efficiency, scalability. Include measurable performance acceptance criteria where possible.
  • Responsive web application: must work across XS (<576px), SM (≥576px), MD (≥768px), LG (≥992px), and XL (≥1200px) viewports. Add responsive requirements for every UI-facing feature.
  • ATDD workflow: every L2 must be testable. Acceptance tests are written first, fail, then implementation makes them pass. Each acceptance test must include a comment identifying which L2(s) it covers.

L1.md format — high-level statements of capability. Plain language, no implementation detail.

# L1 - High-Level Requirements

## L1-001: [Short descriptive title]
[Clear statement of the capability or behavior the system must provide.]

## L1-002: [Short descriptive title]
[Clear statement of the capability or behavior the system must provide.]

L1 guidelines:

  • Each L1 represents a distinct area of functionality or a cross-cutting concern.
  • Plain language a stakeholder (not just a developer) understands.
  • No implementation detail — that belongs in L2.
  • Number sequentially: L1-001, L1-002, …
  • Include cross-cutting L1s for security, performance, and accessibility.

L2.md format — detailed testable specs. Each L2 traces to exactly one L1 and includes Given/When/Then acceptance criteria.

# L2 - Detailed Requirements

## L2-001: [Short descriptive title]
**Traces to:** L1-001

[Detailed description of the specific behavior or constraint.]

**Acceptance Criteria:**
1. Given [precondition], when [action], then [expected result]
2. Given [precondition], when [action], then [expected result]

---

## L2-002: [Short descriptive title]
**Traces to:** L1-001

[Detailed description of the specific behavior or constraint.]

**Acceptance Criteria:**
1. Given [precondition], when [action], then [expected result]

---

L2 guidelines:

  • Each L2 traces to exactly one L1 via the Traces to: field. Multiple L2s can trace to the same L1.
  • Acceptance criteria use Given/When/Then so they translate directly to tests.
  • Be specific enough that two different developers would write the same test.
  • Include boundary conditions, error cases, and security constraints.
  • Number sequentially: L2-001, L2-002, …
  • Include responsive behavior in acceptance criteria where relevant — e.g., "Given viewport width < 576px, when the user views the dashboard, then the layout switches to single-column."

Acceptance test traceability. When acceptance tests are later written (in BI1 / FI1), each test file includes a header comment identifying which L2(s) it covers. Examples:

// Acceptance Test
// Traces to: L2-001, L2-003
// Description: Verify user login with valid credentials and session creation

describe('User Authentication', () => {
  // ...
});

# Acceptance Test
# Traces to: L2-007
# Description: Verify API rate limiting returns 429 after threshold

class TestRateLimiting:
    # ...

The trace comment is not optional — it is how coverage is verified.

Workflow for this task:

  1. Read ./docs/idea.md. Identify features, behaviors, constraints.

  • mvp-backend (MB1, MB2) runs in parallel with mvp-frontend (MF1, MF2) once R2 is done.

  • Deployment (DP1) waits for both BI1 and FI1.

  • Evaluation gates follow every produced artifact (mocks, requirements, MVPs, plans, task lists, implementations).

  • Azure deployment uses the cheapest viable plan provisioned via Azure CLI.
